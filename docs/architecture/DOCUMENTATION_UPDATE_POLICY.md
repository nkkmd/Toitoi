# ドキュメント更新情報ポリシー

**Status: evolving** | **Last updated: 2026-05-19**

## 目的

Toitoi の文書群に含まれる更新情報を、用途ごとにぶれなく管理するための基準を定めます。

この文書は方針文書であり、固定版として扱いません。必要に応じて更新し、変更履歴は Git の履歴で追跡します。

このポリシーの狙いは次の 3 つです。

- 仕様の互換性を明確にする
- 実装ガイドやロードマップを「古い更新情報のまま固定しない」
- どの文書を正本として参照すべきかを分かりやすくする

---

## 基本方針

1. Markdown 文書は、文書単体の `Version` を持たない
2. 文書ヘッダは `status` と `last updated` を正本として扱う
3. 仕様の機械可読な識別情報は、必要な場合のみ JSON Schema やデータ構造内の `schemaVersion` などで管理する
4. 1 つの文書の中に、複数の版の責務を混ぜない
5. 古い内容を残す必要がある場合は archive か別文書に分ける

---

## 文書の分類

### A. 仕様文書

対象は、外部実装や他プロトコルと互換性を持つ必要があるものです。

- `schemas/*.json`
- `docs/protocols/*.md`
- `docs/concepts/TOITOI_VOCABULARY.md`

運用ルール:

- JSON などの機械可読な成果物では `schemaVersion` や `version` を明示する
- Markdown 文書そのものには `Version` を置かない
- 変更履歴が必要な場合は本文冒頭または末尾に簡潔に残す
- 破壊的変更は新しい文書版ではなく、別文書化や archive 化で扱う

### B. 実装ガイド

対象は、開発・実装のための参照文書です。

- `apps/*/*.md`

運用ルール:

- 文書単体の `Version` は置かない
- `status` と `last updated` を本文の冒頭 1 か所に置く
- 実装の現状が変わったら、更新情報ではなく前提条件を更新する

### C. 進行管理文書

対象は、方針・計画・判断基準をまとめる文書です。

- `docs/roadmap/*.md`
- 必要に応じて `docs/architecture/*.md` の判断記録

運用ルール:

- 原則として固定版を前提にしない
- `last updated` を優先する
- 変更履歴が必要な場合は、簡潔な決定ログを残す

### D. 論説・背景文書

対象は、思想・背景・説明を伝える文書です。

- `docs/essays/*.md`

運用ルール:

- 文書単体の `Version` は持たない
- 必要な場合のみ、執筆時点の前提を本文に明記する

---

## 文書単体の版が不要な理由

Markdown 文書の役割は、現時点の判断・手順・概念を伝えることにあり、文書単体の版番号を付けるよりも `status` と `last updated` の方が運用に合っています。

次のような変更は、文書単位の版番号ではなく、本文更新や別文書化で扱います。

- フィールド名やタグ名が変わる
- 必須・任意の条件が変わる
- 解釈ルールが互換性なく変わる
- 既存実装がそのままでは動かなくなる

---

## 更新情報の置き方

推奨は次の通りです。

- JSON Schema は `schemaVersion` を持つ
- Markdown 文書の冒頭は `Status` と `Last updated` に統一する
- 語彙リストやプロトコル文書も、文書単体の `Version` は持たない
- 仕様差分が必要なら本文の変更履歴か別ファイルで管理する

文書内に版番号を散らさず、更新情報は 1 か所に集約してください。

---

## 更新フロー

1. 仕様変更を提案する
2. 互換性影響を確認する
3. 本文や関連文書を更新する
4. 必要なら archive に残す
5. 参照先リンクを更新する

---

## ファイル別の扱い

### `schemas/canonical-event.schema.json`

機械可読な契約なので、`schemaVersion` を正本として扱います。

### `docs/protocols/NOSTR_INQUIRY_SCHEMA.md`

transport 仕様ですが、Markdown 文書の版番号は持たず、必要な差分は本文内の記述と履歴で管理します。

### `docs/concepts/TOITOI_VOCABULARY.md`

語彙は実装や運用に合わせて増えうるため、更新履歴を残しつつ `Status` と `Last updated` で管理します。

### `apps/edge-ai/README.md`

実装ガイドとして扱い、文書単体の版番号は使いません。仕様契約そのものはここでは定義しません。

### `docs/roadmap/IMPLEMENTATION_PLAN.md`

計画文書として扱い、`Status` と `Last updated` を中心に管理します。

---

## 推奨テンプレート

必要に応じて、各文書の冒頭を次のように統一してください。

```md
# Title

**Status: draft / evolving / stable** | **Last updated: YYYY-MM-DD**

Short purpose statement.
```

計画文書では次のような形でも構いません。

```md
# Title

**Status: evolving** | **Last updated: 2026-05-19**
```

---

## 迷ったときの判断

迷ったら、次の順番で考えてください。

1. これは他の実装者が依存する契約か
2. これは今後も互換性を保つ必要があるか
3. これは現時点の運用メモに近いか

1 か 2 に当てはまるなら仕様文書として別管理し、3 に近いなら進行管理文書として扱ってください。
