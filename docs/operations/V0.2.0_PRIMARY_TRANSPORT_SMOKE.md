# v0.2.0 Primary Transport Operational Smoke

**Status: active** | **Target: v0.2.0** | **Last updated: 2026-07-13**

## Purpose

This smoke contract verifies that Toitoi's two primary transport publication paths can emit a Canonical Event, retrieve the resulting transport object, and feed it back through the corresponding ingest pipeline.

The default workspace test is deterministic and does not contact a production relay or carrier.

## Covered paths

### Nostr

1. Convert a Canonical Event to a Nostr draft.
2. Finalize and publish the event through an injected relay publisher.
3. Capture the published relay event.
4. Re-ingest it through `ingestNostrEvents`.
5. Verify inquiry text and Nostr provenance.

### Lingonberry

1. Convert a Canonical Event to a Lingonberry knowledge object.
2. Publish it through the HTTP carrier client.
3. Retrieve the stored object through `getObject`.
4. Re-ingest it through `ingestLingonberryEvents`.
5. Verify inquiry text and Lingonberry provenance.

## Test location

- `packages/protocol/test_primary_transport_operational_smoke.js`

## Network boundary

The CI smoke test substitutes an in-memory publisher for Nostr and a deterministic `fetch` implementation for Lingonberry. This exercises the real converters, outbound functions, HTTP request construction, and ingest pipelines while avoiding production credentials and external network instability.

Actual live checks remain explicit and environment-gated:

- Lingonberry: `LINGONBERRY_LIVE_SMOKE_TEST=1`
- Existing multi-transport live check: `TOITOI_LIVE_MULTI_TRANSPORT_TEST=1`

Live checks are operational confirmation tools, not default unit-test dependencies.

## Release interpretation

Passing this test means that the primary transport runtime boundaries have no known deterministic regression in the publish/retrieve/re-ingest path. It does not certify production relay availability, carrier availability, authentication policy, rate limiting, or long-duration reliability.
