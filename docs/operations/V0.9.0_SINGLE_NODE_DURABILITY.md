# Toitoi v0.9.0 Single-node Durability Profile

## Purpose

This profile defines the minimum supported production-like deployment for small-group operation. It deliberately remains single-node while making durability, recovery, and failure boundaries explicit.

## Source-of-truth hierarchy

1. Canonical append-only storage
2. raw transport storage referenced by provenance
3. reviewed workflow and audit records
4. durable AI and delivery job state
5. rebuildable search, metrics, and presentation indexes

Derived indexes must never become the only copy of user data or review decisions.

## Required filesystem layout

```text
TOITOI_DATA_DIR/
├── canonical/
├── raw/
├── workflow/
├── queue/
├── audit/
├── indexes/
├── migrations/
└── backups/
```

Deployments may map these paths to separate volumes, but backup and restore must preserve their declared relationships.

## Durable jobs

Every job record must contain:

- stable job ID
- job type and schema version
- canonical source references
- status: `queued`, `running`, `succeeded`, `failed`, or `dead_letter`
- attempt count and maximum attempts
- created, started, completed, and next-attempt timestamps
- last error code and message
- deterministic idempotency identity

A process restart must convert abandoned `running` jobs to retryable state according to an explicit lease timeout. Exceeding the attempt limit moves a job to dead-letter state without deleting its source record.

## Graceful shutdown

On `SIGTERM` or `SIGINT`, the process must:

1. stop accepting new mutable work
2. report readiness as false
3. stop claiming new jobs
4. complete or checkpoint in-flight work within the configured deadline
5. persist final job and delivery states
6. flush structured logs and metrics
7. close storage handles

Forced termination after the deadline must remain recoverable through lease expiry and replay.

## Backup

A backup contains a manifest with:

- backup format version
- application version
- schema and migration versions
- creation timestamp
- included paths
- file sizes and checksums
- highest audit sequence and hash
- queue counts by state
- canonical record count

The backup sequence is:

1. enter maintenance or consistent-snapshot mode
2. pause job claims and mutable requests
3. flush storage handles
4. copy source-of-truth stores
5. record checksums and manifest
6. resume operation

Derived indexes may be omitted when the manifest declares they will be rebuilt.

## Restore

Restore validation must:

1. verify manifest version and checksums
2. restore into an empty target directory
3. apply required forward migrations
4. verify the audit hash chain
5. recover queue leases and dead-letter state
6. replay Canonical storage
7. rebuild derived indexes
8. compare record counts and deterministic fixture queries
9. enable readiness only after all mandatory checks pass

## Migrations

Migrations are ordered, immutable records. Tooling must support:

- `plan`: show pending migrations
- `dry-run`: validate prerequisites without mutation
- `apply`: execute each migration once
- `status`: show current and target versions

A migration that changes observable Canonical semantics requires the governance procedure and compatible Conformance Suite fixtures.

## Health

- liveness means the process can serve a response
- readiness means mandatory storage, queue, schema, and migration checks pass
- optional transport outages degrade delivery status but do not necessarily fail core readiness
- source-of-truth storage corruption always fails readiness

## Structured logs and metrics

Every structured log should include timestamp, level, component, request or job ID, actor ID when applicable, action, outcome, and stable error code.

Minimum metrics:

- requests and errors by route and code
- authorization denials
- rate-limit denials
- idempotency replays and conflicts
- queue depth by state
- job attempts and dead-letter count
- audit append failures
- transport delivery status
- backup and restore outcome
- replay and index rebuild duration

## Recovery drills

Before v0.9.0 release, deterministic fixtures must cover:

- termination during a running job
- dead-letter transition and operator retry
- restore without derived indexes
- interrupted migration before commit
- unavailable transport with successful Canonical persistence
- audit-chain verification failure
