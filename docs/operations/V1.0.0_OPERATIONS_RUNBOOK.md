# Toitoi v1.0.0 Minimum Operations Runbook

**Status: release candidate**  
**Language status:** English and Japanese sections are maintained as equivalent.  
**Last synchronized:** 2026-07-23

## Purpose

This runbook defines the minimum safe operating procedure for the Toitoi v1.0.0 reference implementation. It covers a single-node deployment using persistent local storage, the reference API process, deterministic recovery tooling, and optional external transports and AI providers.

This is not a production certification. Deployment-specific process supervision, TLS termination, OAuth/OIDC, distributed queues, and managed secret stores remain operator responsibilities.

## Operational invariants

1. Canonical storage is authoritative.
2. Raw storage, Canonical storage, durable queue state, and audit records are durable state.
3. Search indexes, facets, metrics, and health projections are rebuildable derived state.
4. Transport failure does not invalidate Canonical persistence.
5. AI failure does not block observation persistence.
6. AI annotation acceptance is not publication approval.
7. Authentication identity, actor identity, transport identity, and Canonical Event identity remain distinct.
8. Backup and restore procedures must preserve auditability and stable Canonical IDs.

## Prerequisites

- Git
- Node.js compatible with repository CI
- Corepack
- pnpm declared by the root `packageManager`
- writable persistent directories
- an operator-controlled location for credentials and secrets

## Persistent layout

Recommended reference layout:

```bash
mkdir -p "$HOME/.local/share/toitoi/storage"
mkdir -p "$HOME/.local/share/toitoi/ai"
mkdir -p "$HOME/.local/share/toitoi/search"
mkdir -p "$HOME/.local/share/toitoi/backups"
```

Keep persistent data outside the repository working tree.

Durable state:

- raw storage;
- Canonical storage;
- durable queue state;
- audit records;
- operator-managed configuration and secrets.

Rebuildable state:

- FTS5 search index;
- context facets;
- related-inquiry candidates;
- metrics and health projections.

## Installation and verification

```bash
git clone https://github.com/nkkmd/Toitoi.git
cd Toitoi
corepack enable
corepack pnpm install --frozen-lockfile
corepack pnpm test
```

Do not continue with release deployment when the complete workspace test fails.

## Start

Development-compatible reference mode:

```bash
TOITOI_STORAGE_DIR="$HOME/.local/share/toitoi/storage" \
TOITOI_AI_STORAGE_DIR="$HOME/.local/share/toitoi/ai" \
TOITOI_SEARCH_INDEX_FILE="$HOME/.local/share/toitoi/search/toitoi.sqlite" \
TOITOI_PROTOCOL=nostr \
corepack pnpm --filter @toitoi/api start
```

Authenticated reference mode:

```bash
TOITOI_AUTH_REQUIRED=true \
TOITOI_STORAGE_DIR="$HOME/.local/share/toitoi/storage" \
TOITOI_AI_STORAGE_DIR="$HOME/.local/share/toitoi/ai" \
TOITOI_SEARCH_INDEX_FILE="$HOME/.local/share/toitoi/search/toitoi.sqlite" \
TOITOI_PROTOCOL=nostr \
corepack pnpm --filter @toitoi/api start
```

Reference authentication headers:

- `X-Toitoi-Actor-Id`
- `X-Toitoi-Roles`
- `Idempotency-Key` for mutations

The header-based provider is a deterministic reference boundary, not OAuth/OIDC certification.

## Health checks

Check process liveness:

```bash
curl -fsS http://127.0.0.1:3000/health/live
```

Check readiness:

```bash
curl -fsS http://127.0.0.1:3000/health/ready
```

A live process may still be unready. Do not direct mutations to an unready instance.

## Stop and restart

Use the process supervisor for the deployment when available. For a foreground reference process, send an ordinary termination signal and allow graceful shutdown to complete.

Before restart:

1. confirm no backup or migration is actively writing;
2. preserve logs and request IDs for any incident;
3. confirm the persistent paths are unchanged;
4. restart with the same authoritative storage roots;
5. recheck liveness and readiness.

Do not replace persistent directories with an empty path merely to make the process start.

## Backup

Back up all durable state together. Do not treat the FTS5 file as the only backup.

The reference recovery implementation records:

- relative path;
- file size;
- SHA-256 digest;
- manifest version;
- creation time.

Run the operations validation before relying on a backup procedure:

```bash
corepack pnpm --filter @toitoi/operations test
```

For an operational backup:

1. identify every durable file and directory;
2. quiesce writes or use a deployment-specific consistent snapshot mechanism;
3. copy durable state to a dated backup root;
4. create and retain a checksum manifest;
5. verify the manifest against the backup copy;
6. store configuration and secret references separately under operator control;
7. record the application commit and migration version.

## Restore

Restore into a separate root first.

1. stop the target process;
2. preserve the failed or current state for investigation;
3. copy the selected backup into a new restore root;
4. verify every manifest entry and SHA-256 digest;
5. verify expected Canonical IDs and audit records;
6. configure the API to use the restored root;
7. rebuild derived state from Canonical storage;
8. start the process;
9. verify liveness, readiness, search, and representative reads;
10. retain the restore report.

Never declare a restore successful solely because the process starts.

## Migration

Before migration:

1. record the current commit and data version;
2. create and verify a complete backup;
3. review the migration plan;
4. run dry-run mode;
5. confirm no unexpected target version or destructive action appears;
6. apply the migration once;
7. record applied migration names and resulting version;
8. rebuild derived state;
9. run representative contract and recovery checks.

Dry-run must not mutate durable state.

A migration failure is a stop condition. Preserve logs, applied-step records, request IDs, and the pre-migration backup before deciding whether to retry or roll back.

## Queue and dead-letter recovery

When asynchronous jobs fail:

1. inspect the durable queue state;
2. distinguish pending, retryable, exhausted, and dead-letter jobs;
3. retain the original job input and failure information;
4. correct the provider, configuration, or data issue;
5. retry explicitly through the supported queue boundary;
6. confirm that retry does not duplicate Canonical publication or audit effects;
7. preserve dead-letter evidence until the incident is closed.

Do not edit durable queue files manually while the process is running.

## Transport failure and quarantine

Transport states may include pending, delivered, failed, skipped, or quarantined.

When a transport is unavailable:

- keep the Canonical Event stored locally;
- expose the delivery state;
- do not create a new Canonical ID for a retry;
- correct credentials or endpoint configuration;
- retry through the transport runtime;
- verify that external source identifiers remain transport-scoped.

A transport outage must not be reported as loss of the observation or Canonical Event.

## AI provider unavailable or disabled

When the AI provider is unavailable:

- continue accepting and persisting observations;
- keep generation jobs pending or failed according to the queue contract;
- preserve provider, model, prompt, and raw error metadata where applicable;
- do not create approved annotations automatically;
- do not bypass human annotation review or independent publication approval.

Deterministic CI providers and fixtures do not establish live-model reproducibility.

## Credentials and secrets

- keep credentials outside repository fixtures and committed configuration;
- use deployment-specific secret storage;
- rotate compromised or expired credentials;
- restart or reload only the affected boundary where supported;
- verify transport or provider health after rotation;
- record the rotation without recording the secret value;
- quarantine uncertain outbound delivery rather than weakening validation.

## Upgrade

1. read release notes and migration requirements;
2. verify the target commit, tag, and release artifacts;
3. create and verify a durable-state backup;
4. stop or drain the current instance;
5. install dependencies with the frozen lockfile;
6. run tests and strict Conformance input;
7. run migration plan and dry-run;
8. apply migrations;
9. rebuild derived state;
10. start the upgraded instance;
11. verify health, representative reads, workflow authority, queue state, and transport state;
12. retain the upgrade record.

Strict v1 Conformance command:

```bash
corepack pnpm --filter @toitoi/conformance exec \
  toitoi-conformance \
  --input fixtures/reference/v1.0.0/conformance-input.json \
  --output /tmp/toitoi-v1-conformance-report.json \
  --pretty
```

## Rollback

Rollback is an operator decision, not an automatic response to every failure.

Before rollback:

1. determine whether the migration is backward compatible;
2. preserve the failed upgraded state;
3. preserve logs, audit records, migration records, and request IDs;
4. select a verified pre-upgrade backup;
5. restore into a separate root;
6. verify checksums and Canonical IDs;
7. start the previous verified application version against the compatible restored state;
8. rebuild derived state;
9. verify health and representative workflows;
10. record the rollback reason and remaining risk.

Do not run an older application against a newer incompatible durable schema.

## Incident evidence

For every material incident, retain:

- application commit or tag;
- timestamps and timezone;
- request IDs;
- actor IDs and declared roles where appropriate;
- health responses;
- queue and transport states;
- audit-log references;
- migration version;
- backup and restore manifest results;
- corrective actions and unresolved risks.

Do not include secrets or unnecessary sensitive field data in incident reports.

## Operational verification checklist

- [ ] persistent directories exist outside the repository;
- [ ] complete workspace tests pass;
- [ ] liveness and readiness succeed;
- [ ] authenticated mode enforces authority boundaries where enabled;
- [ ] durable and rebuildable state are identified;
- [ ] backup manifest verification succeeds;
- [ ] restore verification succeeds in a separate root;
- [ ] migration plan, dry-run, and apply are recorded;
- [ ] queue retry and dead-letter procedures are understood;
- [ ] transport failure remains separate from Canonical persistence;
- [ ] AI failure remains separate from observation persistence;
- [ ] upgrade and rollback records are retained;
- [ ] known limitations are visible.

# Toitoi v1.0.0 最小運用Runbook

**状態: release candidate**  
**言語状態:** 英語版と日本語版は同等の内容として管理します。  
**最終同期日:** 2026-07-23

## 目的

このrunbookは、Toitoi v1.0.0参照実装を安全に運用するための最小手順を定めます。persistent local storage、reference API process、決定的な復旧tool、任意の外部transportおよびAI providerを使用するsingle-node deploymentを対象とします。

これはproduction認定ではありません。deployment固有のprocess supervision、TLS termination、OAuth/OIDC、distributed queue、managed secret storeはoperatorの責任です。

## 運用上の不変条件

1. Canonical storageが正本です。
2. raw storage、Canonical storage、durable queue state、audit recordはdurable stateです。
3. search index、facet、metrics、health projectionは再構築可能なderived stateです。
4. transport failureはCanonical persistenceを無効化しません。
5. AI failureはobservation persistenceを妨げません。
6. AI annotation acceptanceはpublication approvalではありません。
7. authentication identity、actor identity、transport identity、Canonical Event identityは分離します。
8. backupとrestoreはauditabilityとstable Canonical IDを保持しなければなりません。

## 前提条件

- Git
- repository CIと互換性のあるNode.js
- Corepack
- ルート`packageManager`で宣言されたpnpm
- 書き込み可能なpersistent directory
- operatorが管理するcredential・secret保存先

## Persistent layout

推奨reference layout:

```bash
mkdir -p "$HOME/.local/share/toitoi/storage"
mkdir -p "$HOME/.local/share/toitoi/ai"
mkdir -p "$HOME/.local/share/toitoi/search"
mkdir -p "$HOME/.local/share/toitoi/backups"
```

persistent dataはrepository working treeの外に置きます。

Durable state:

- raw storage;
- Canonical storage;
- durable queue state;
- audit record;
- operator管理のconfigurationとsecret。

Rebuildable state:

- FTS5 search index;
- context facet;
- related-inquiry candidate;
- metricsとhealth projection。

```bash
cd Toitoi
```

development-compatible reference mode:

```bash

TOITOI_SEARCH_INDEX_FILE="$HOME/.local/share/toitoi/search/toitoi.sqlite" \

```

authenticated reference mode:

```bash
TOITOI_AUTH_REQUIRED=true \

TOITOI_SEARCH_INDEX_FILE="$HOME/.local/share/toitoi/search/toitoi.sqlite" \
TOITOI_PROTOCOL=nostr \
```

## Health check

process liveness:

```bash
curl -fsS http://127.0.0.1:3000/health/live
```

readiness:

```bash
curl -fsS http://127.0.0.1:3000/health/ready
```

processがliveでもunreadyの場合があります。unready instanceへmutationを送ってはなりません。

## 停止と再起動

deploymentでprocess supervisorを使用している場合は、その手順に従います。foreground reference processでは通常のtermination signalを送り、graceful shutdownの完了を待ちます。

再起動前:

1. backupまたはmigrationが書き込み中でないことを確認する;
2. incidentに関係するlogとrequest IDを保存する;
3. persistent pathが変わっていないことを確認する;
4.同じauthoritative storage rootで再起動する;
5. livenessとreadinessを再確認する。

processを起動させるためだけにpersistent directoryを空のpathへ置き換えてはなりません。

## Backup

すべてのdurable stateをまとめてbackupします。FTS5 fileだけをbackupとして扱ってはなりません。

reference recovery implementationは次を記録します。

- relative path;
- file size;
- SHA-256 digest;
- manifest version;
- creation time。

backup手順へ依存する前にoperations validationを実行します。

```bash
corepack pnpm --filter @toitoi/operations test
```

運用backup:

1. すべてのdurable fileとdirectoryを特定する;
2. 書き込みを停止するか、deployment固有のconsistent snapshotを使用する;
3. dated backup rootへdurable stateをcopyする;
4. checksum manifestを作成・保存する;
5. backup copyに対してmanifestを検証する;
6. configurationとsecret referenceをoperator管理下で別途保存する;
7. application commitとmigration versionを記録する。

## Restore

最初に別rootへrestoreします。

1. target processを停止する;
2. 調査のため現在または失敗状態を保存する;
3. 選択したbackupを新しいrestore rootへcopyする;
4. すべてのmanifest entryとSHA-256 digestを検証する;
5. expected Canonical IDとaudit recordを確認する;
6. restored rootを使用するようAPIを設定する;
7. Canonical storageからderived stateを再構築する;
8. processを起動する;
9. liveness、readiness、search、代表的readを検証する;
10. restore reportを保存する。

processが起動したことだけでrestore成功と判断してはなりません。

## Migration

migration前:

1. 現在のcommitとdata versionを記録する;
2. 完全なbackupを作成・検証する;
3. migration planを確認する;
4. dry-runを実行する;
5. 想定外のtarget versionや破壊的actionがないことを確認する;
6. migrationを一度だけapplyする;
7. applied migration名とresulting versionを記録する;
8. derived stateを再構築する;
9. representative contract・recovery checkを実行する。

dry-runはdurable stateを変更してはなりません。

migration failureは停止条件です。retryまたはrollbackを判断する前に、log、applied-step record、request ID、pre-migration backupを保存します。

## Queueとdead-letter recovery

asynchronous jobが失敗した場合:

1. durable queue stateを確認する;
2. pending、retryable、exhausted、dead-letterを区別する;
3. original job inputとfailure informationを保存する;
4. provider、configuration、data issueを修正する;
5. supported queue boundaryから明示的にretryする;
6. retryがCanonical publicationやaudit effectを重複させないことを確認する;
7. incident終了までdead-letter evidenceを保存する。

process稼働中にdurable queue fileを手動編集してはなりません。

## Transport failureとquarantine

transport stateにはpending、delivered、failed、skipped、quarantinedが含まれます。

transportが利用できない場合:

- Canonical Eventをlocalに保持する;
- delivery stateを表示する;
- retryのために新しいCanonical IDを作成しない;
- credentialまたはendpoint configurationを修正する;
- transport runtimeからretryする;
- external source identifierがtransport-scopedのままであることを確認する。

transport outageをobservationまたはCanonical Eventの消失として報告してはなりません。

## AI providerが利用不能または無効な場合

- observationの受付・保存を継続する;
- generation jobをqueue contractに従いpendingまたはfailedとして保持する;
- 該当する場合、provider、model、prompt、raw error metadataを保持する;
- approved annotationを自動作成しない;
- human annotation reviewまたはindependent publication approvalを迂回しない。

deterministic CI providerとfixtureはlive-model reproducibilityの証明ではありません。

## Credentialとsecret

- credentialをrepository fixtureやcommit済みconfigurationに保存しない;
- deployment固有のsecret storageを使用する;
- compromiseまたはexpirationが発生したcredentialをrotationする;
- 対応可能な場合、影響を受けたboundaryだけをrestartまたはreloadする;
- rotation後にtransportまたはprovider healthを検証する;
- secret valueを記録せずrotationを記録する;
- validationを弱めるのではなく、不確実なoutbound deliveryをquarantineする。

## Upgrade

1. release notesとmigration requirementを読む;
2. target commit、tag、release artifactを検証する;
3. durable-state backupを作成・検証する;
4. current instanceを停止またはdrainする;
5. frozen lockfileでdependencyをinstallする;
6. testとstrict Conformance inputを実行する;
7. migration planとdry-runを実行する;
8. migrationをapplyする;
9. derived stateを再構築する;
10. upgraded instanceを起動する;
11. health、representative read、workflow authority、queue state、transport stateを検証する;
12. upgrade recordを保存する。

strict v1 Conformance command:

```bash
corepack pnpm --filter @toitoi/conformance exec \
  toitoi-conformance \
  --input fixtures/reference/v1.0.0/conformance-input.json \
  --output /tmp/toitoi-v1-conformance-report.json \
  --pretty
```

## Rollback

rollbackはoperator判断であり、すべてのfailureに対する自動対応ではありません。

rollback前:

1. migrationにbackward compatibilityがあるか判断する;
2. failed upgraded stateを保存する;
3. log、audit record、migration record、request IDを保存する;
4. verified pre-upgrade backupを選択する;
5. 別rootへrestoreする;
6. checksumとCanonical IDを検証する;
7. compatibleなrestored stateに対して以前のverified application versionを起動する;
8. derived stateを再構築する;
9. healthとrepresentative workflowを検証する;
10. rollback reasonとremaining riskを記録する。

古いapplicationを互換性のない新しいdurable schemaに対して実行してはなりません。

## Incident evidence

重大incidentごとに次を保存します。

- application commitまたはtag;
- timestampとtimezone;
- request ID;
- 必要に応じてactor IDとdeclared role;
- health response;
- queueとtransport state;
- audit-log reference;
- migration version;
- backup・restore manifest result;
- corrective actionとunresolved risk。

incident reportへsecretや不要な機微field dataを含めてはなりません。
