## Findings and resolutions

### 1. Repository release version and wire-schema version

**Finding:** The repository release is `v1.0.0`, while Canonical Events use `schemaVersion: "0.1.0"`.

**Resolution:** Retain `0.1.0` as the stable wire-schema identifier. Do not rewrite append-only historical records merely to align numbers. The distinction is now explicit in:

- `schemas/canonical-event.schema.json`
- `docs/reference/V1_CONTRACT_INDEX.md`
- schema contract tests

A future wire-schema change requires reader/writer rules, migration fixtures, compatibility documentation, and Conformance coverage.

### 2. lineage relation spelling

**Finding:** The schema guidance contained the advisory spelling `synthesis`, while the implemented derivation relation is `synthesizes`.

**Resolution:** The normative relation vocabulary is synchronized with `packages/protocol/derived_inquiry.js`:

- `derived_from`
- `translated_from`
- `observed_alongside`
- `contrasts_with`
- `synthesizes`
- `reframes`
- `annotates`
- `revises`

`synthesis` remains an event `type`; it is not the normative lineage relation spelling.

### 3. Canonical extension behavior

**Finding:** The v1 freeze checklist required explicit unknown-field behavior.

**Resolution:** The contract now distinguishes:

- forward-compatible unknown top-level fields: tolerated;
- `contexts` and `meta`: extensible;
- `body`, `provenance`, `rawRef`, and other closed nested objects: unknown fields rejected by schema.

Schema-level assertions now lock this behavior.

### 4. Native v1 versus legacy compatibility

**Finding:** Earlier Conformance logic risked treating old `content`-based and legacy-ID fixtures as native v1.

**Resolution:** Native v1 validation is strict. Legacy shapes are accepted only through the explicit `v0.9.0` compatibility profile. Passing the legacy profile does not establish native v1 Canonical Event conformance.

### 5. Human authority boundaries

**Reviewed result:** AI annotation acceptance, relation confirmation, publication approval, moderation, and operator authority remain separate auditable decisions.

The reference fixture and offline Golden Path verify that annotation review and publication approval are independently attributable.

### 6. Identity boundaries

**Reviewed result:** Authentication identity, actor identity, transport identity, and Canonical Event identity remain distinct.

Similarity, vocabulary mapping, transport projection, and semantic relation do not merge Canonical identity automatically.

### 7. Persistence and projection boundaries

**Reviewed result:** Canonical storage is authoritative. Search indexes, facets, queue indexes, metrics, and health views are rebuildable derived state.

Transport failure changes delivery state and does not invalidate Canonical persistence.

### 8. Provenance and raw data

**Reviewed result:** `provenance.sources` records source involvement. Top-level `rawRef` points to separately controlled raw material. Embedded raw payloads are rejected by Conformance checks.

Raw observation capture does not imply publication.

### 9. Replay and recovery

**Reviewed result:** Deterministic tests cover:

- reverse-order replay search equivalence;
- Canonical ID preservation;
- backup manifest and SHA-256 verification;
- restore to a separate root;
- corruption detection;
- migration planning;
- dry-run and apply;
- rebuildable derived state.

### 10. External Conformance boundary

**Reviewed result:** The CLI accepts fixed JSON input and emits a machine-readable report without importing private application modules from the fixture.

Conformance indicates observable interoperability only. It does not establish governance authority, deployment security, live-service availability, or agronomic correctness.

## Compatibility decisions

- Canonical Event `schemaVersion: "0.1.0"` is retained for the repository v1 release.
- Native v1 writers should emit `body`, not legacy `content`.
- Native v1 Canonical IDs use `tt:evt:<opaque-id>`.
- Legacy fixture support requires the explicit `v0.9.0` compatibility profile.
- Unknown top-level fields are tolerated; closed nested objects remain closed.
- Existing append-only records are not silently rewritten.
