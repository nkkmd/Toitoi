# Toitoi v1.0.0 Reference Scenario

**Status: implementation reference** | **Fixture: `fixtures/reference/v1.0.0/east-side-weed-scenario.json`**

## Scenario

A grower records the following field observation:

> 畑の東側だけ雑草の種類が違う。東側にはスゲに似た草が多く、西側には少ない。

Toitoi preserves the original local wording and context, proposes inquiry candidates asynchronously, requires explicit human review, publishes an approved Canonical Event, connects it to a related inquiry from another region, and supports a human-confirmed synthesis without merging either source identity.

## Golden Path

### 1. Offline observation capture

The observation is stored locally before synchronization. The raw observation retains:

- Japanese source text;
- local terms such as `東べり` and `スゲに似た草`;
- field edge, climate, soil, crop, and seasonal context;
- a `local-capture` provenance source;
- a separate `rawRef` pointing to the raw fixture record.

Offline capture does not imply synchronization or publication.

### 2. Explicit synchronization and asynchronous AI work

After an explicit synchronization action, the stored observation becomes eligible for asynchronous inquiry generation. The deterministic reference provider returns multiple candidates and records:

- provider and model identifiers;
- model and prompt versions;
- raw candidate output;
- source observation ID;
- job and annotation IDs.

AI failure must not roll back the already persisted observation.

### 3. Human annotation review

A reviewer selects one candidate and edits it to include soil moisture, sunlight, and microtopography:

> 畑の東側だけ土壌水分、日照、微地形の条件が異なり、雑草の種類に差が出ているのはなぜか？

The annotation decision is `accept-with-edit`. This decision accepts an AI proposal for Draft promotion; it is not publication approval.

### 4. Inquiry Draft and publication review

The accepted annotation is promoted to an Inquiry Draft. The Draft retains references to both the source observation and reviewed annotation.

A publisher independently reviews the Draft. The fixture intentionally records different actors for annotation review and publication approval so the two decisions remain independently attributable.

### 5. Canonical persistence

After approval, the inquiry is persisted as `tt:evt:v1-inquiry-east-side-weeds`.

The published event retains:

- the human-edited inquiry text;
- original context and local terms;
- `derived_from` lineage to the observation;
- workflow provenance;
- annotation and publication review metadata.

Canonical persistence is authoritative even when one or more transports are unavailable.

### 6. Multi-transport projection

The deterministic fixture records:

- Nostr: `delivered`;
- Lingonberry: `delivered`;
- ATProto: `quarantined` because credentials are deliberately unavailable in deterministic CI.

All transport records reference the same Canonical Event ID. A quarantined transport projection does not invalidate the stored event.

### 7. Discovery across contexts

A related inquiry from another region asks why sedges appear along a wetter eastern field edge. It retains its own Canonical ID, English wording, ATProto provenance, regional context, soil, crop, and local terms.

Search and context exploration may classify the inquiries as related candidates. Similarity does not establish identity.

### 8. Vocabulary mapping

The fixture preserves local wording and adds reviewed mapping claims:

- `東べり` → `field-edge`;
- `スゲに似た草` → `sedges`, with uncertainty retained.

A mapping claim is a reviewed semantic bridge, not an identity rewrite or deletion of the local term.

### 9. Human-confirmed derivation

A new inquiry synthesizes the two source inquiries:

> 異なる地域の圃場東端で、土壌水分と微地形はスゲ類の出現差にどう関係するか？

The derived inquiry has its own Canonical ID and two `synthesizes` lineage edges. It does not replace or merge either source inquiry.

### 10. Replay and derived-index rebuild

The reference recovery contract treats the following as durable:

- raw storage;
- Canonical storage;
- durable queue state;
- audit log.

The following remain rebuildable:

- search index;
- context facets;
- related-inquiry candidates.

Rebuilding from a different input order must produce equivalent deterministic search results.

## Deterministic validation

The default workspace test command discovers the v1 tests automatically:

```bash
corepack pnpm test
```

The v1 reference assets currently validate:

- Canonical Event shape and unique IDs;
- observation, annotation, Draft, and publication references;
- separation of annotation review and publication approval;
- multi-source synthesis lineage;
- transport status without Canonical invalidation;
- confirmed vocabulary mappings;
- recovery coverage;
- FTS5 search and context facets;
- deterministic search equivalence after replay rebuild.

Default CI must not require a live model, relay, PDS, or external network.

## Expected UI states

The mobile/offline walkthrough must expose, where applicable:

`queued → syncing → failed/retryable → reviewed → approved → published → delivered/skipped/quarantined`

The UI must also require confirmation of sensitive information and publication scope before synchronization and publication.

## Safety interpretation

This scenario demonstrates traceability and workflow behavior. It does not claim that the generated inquiry, vocabulary mapping, or cross-region comparison is agronomically correct. Participants and operators must be able to edit, reject, withdraw, correct, and republish through the documented human and governance boundaries.
