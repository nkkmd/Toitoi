## Issueごとの責務境界

### #16 lineage tree

読み取りと可視化を担当します。lineage の作成・公開処理は #18 の責務です。

### #17 context exploration

contexts / relationships を使った発見と比較を担当します。semantic similarity による identity merge は行いません。

### #18 derived inquiry creation

既存 Inquiry Draft、human review、publication guard を使い、lineage / provenance を保持した派生 inquiry の作成と公開を担当します。

### #19 release tracking

個別実装ではなく、横断 acceptance criteria、回帰確認、文書、release gate を管理します。

## 初期 relation / lineage type

v0.3.0 の派生操作では、次を正式な初期対象とします。

- `derived_from`
- `translated_from`
- `annotates`
- `reframes`
- `revises`
- `synthesizes`

次は文脈・関係表示には利用できますが、派生操作UIの必須対象から外します。

- `observed_alongside`
- `contrasts_with`

## Issue追加ルール

実装中に作業を分割する場合も、次の条件を守ります。

- #16〜#18 の acceptance criteria のどれを満たすIssueか明示する
- transport 固有の作業は、canonical / API / UI の利用者価値と接続する
- 条件付きスコープを必須スコープより先に blocker にしない
- 新transport、graph inference、大規模最適化を v0.3.0 に混入させない
