## 1. リリースの位置づけ

v0.3.0 は、v0.2.0 で固定した「問いの一生」の Golden Path を、閲覧可能な reference implementation から、問いを発見・比較・派生できる最初の知識空間へ進めるリリースです。

このリリースでは新しい transport や抽象層の追加を主目的にしません。Canonical Event、multi-transport、canonical identity、provenance、replay、Standard API、Inquiry Draft、human review、publication guard という既存基盤を維持しながら、Toitoi 固有の利用体験を完成させます。

中心テーマは次の三つです。

1. 問いの系統樹
2. 文脈に基づく inquiry 探索
3. lineage を保持した派生 inquiry 作成

---

## 2. リリース目標

> 問いを閲覧する基盤から、問い同士を発見・比較・派生できる「問いが育つ知識空間」へ進化させる。

雑草相 Golden Path を使い、利用者が次の一連の操作をブラウザ上で実行できる状態をリリース基準とします。

```text
inquiry detail を読む
  ↓
系譜を辿る
  ↓
context から関連 inquiry を探す
  ↓
派生元と relation type を選ぶ
  ↓
Inquiry Draft を確認・編集する
  ↓
人間が承認する
  ↓
Canonical Event として公開する
  ↓
系統樹と provenance に反映される
```

## 3. 必須スコープ

### 3.1 問いの系統樹

追跡 Issue: [#16](https://github.com/nkkmd/Toitoi/issues/16)

- Standard API の lineage response を利用した系譜ビュー
- root / parent / child / derived inquiry の区別
- relation / lineage type の表示
- inquiry detail と系譜ビューの相互遷移
- node 選択による inquiry detail 表示
- loading / empty / ready / error state
- provenance 概要の表示
- 欠損参照・循環参照に対する防御
- Golden Path fixture による frontend contract / E2E 検証

### 3.2 文脈探索

追跡 Issue: [#17](https://github.com/nkkmd/Toitoi/issues/17)

- context / relationship query の Standard API 契約確認と必要な補強
- context 条件による inquiry 一覧
- inquiry detail から関連 context への遷移
- 選択中の検索条件の明示
- 異なる地域・文脈の inquiry の比較表示
- empty / invalid query / error state
- 類似・関連と canonical identity を混同しない表示
- Golden Path fixture による API / frontend contract 検証

### 3.3 派生 inquiry 作成

追跡 Issue: [#18](https://github.com/nkkmd/Toitoi/issues/18)

- 派生元 inquiry の選択
- relation / lineage type の選択
- Inquiry Draft と human review workflow の再利用
- Canonical Event 生成時の lineage / provenance 保持
- publication guard の維持
- 公開後の lineage tree への反映
- transport publish → retrieve → re-ingest の回帰確認

v0.3.0 で正式に操作対象とする relation / lineage type は次の範囲に固定します。

```text
derived_from
translated_from
annotates
reframes
revises
synthesizes
```

`observed_alongside` と `contrasts_with` は探索・表示上の relationship として扱えますが、派生操作の必須UIには含めません。

## 4. 横断要件

### 4.1 Human-in-the-loop

- AI生成または利用者入力から作られた inquiry は、公開前に Inquiry Draft を経由する
- `approved` 以外の draft は公開しない
- 採用・修正・却下の状態を追跡可能にする
- AIが関与した場合も無人公開を行わない

### 4.2 Canonical semantics

- frontend は transport 固有 event ではなく Standard API の canonical view を利用する
- semantic similarity を canonical identity の根拠にしない
- ambiguity がある inquiry は自動統合せず、relation / lineage として保持する
- Canonical Event contract に未文書化の破壊的変更を加えない

### 4.3 Provenance / replay

- 派生元、操作種別、作成主体、AI関与、人間確認を追跡可能にする
- raw / canonical storage から lineage を再構築できる
- live ingest と replay で同じ canonical view を生成する
- Nostr / Lingonberry の再取り込みで lineage と provenance を失わない

### 4.4 UI accessibility and resilience

- スマートフォン幅を含む基本的なレスポンシブ表示
- keyboard だけでも主要操作へ到達できる構造
- loading / empty / ready / error state を明示する
- 一部データ欠損が画面全体の破綻につながらない

---

## 5. 条件付きスコープ

次の項目は、必須スコープを壊さず、小さな契約として実装可能な場合に限り v0.3.0 に含めます。

- lightweight な類似 inquiry 候補
- identity state (`exact`, `claimed`, `verified`, `probable`, `related`, `unknown`) の読み取り専用表示
- AI による relation 候補の提示
- provenance card の詳細表示改善

条件付きスコープは release blocker にしません。

---

## 6. 非対象

v0.3.0 では次を固定しません。

- 新しい transport の追加
- embeddings を必須とする意味検索
- graph inference
- 大規模 graph / database 最適化
- 本格的なオフライン同期
- 語彙管理UI
- relation type の自由編集UI
- 複数人同時編集
- AIによる無人派生・無人公開
- 外部SDKの完成
- TIP / governance process の完成
- production-grade authentication / authorization / rate limiting

## 7. Acceptance Criteria

### 7.1 User journey

- 雑草相 Golden Path の inquiry detail を表示できる
- detail から三段階以上の lineage を辿れる
- lineage node から別 inquiry の detail を開ける
- context を一つ以上指定して関連 inquiry を取得できる
- 関連 inquiry を派生元として draft を作成できる
- relation type を選び、draft を編集できる
- human review で承認した draft だけを公開できる
- 公開された派生 inquiry が lineage tree に現れる
- provenance から派生元・作成主体・人間確認状態を確認できる

### 7.2 Contract and regression

- workspace root の `corepack pnpm install` が成功する
- workspace root の `corepack pnpm test` が成功する
- Golden Path integration E2E が成功する
- Inquiry Draft / publication guard の contract test が成功する
- Standard API / frontend contract test が成功する
- lineage / context / derived inquiry の新規テストが default CI に含まれる
- Nostr / Lingonberry deterministic operational smoke が成功する
- canonical identity / provenance / replay に既知回帰がない

### 7.3 Documentation

- frontend の操作方法と画面境界が `apps/frontend/README.md` にある
- Standard API の追加・変更点が `apps/api/README.md` と architecture docs に反映されている
- relation / lineage type の意味が正本文書で説明されている
- release notes と release runbook が最終化されている
- 既知制約と非対象が公開文書に記載されている

## 8. 実装順序

依存関係を考慮し、次の順序を推奨します。

### Step 1: 契約の固定

- lineage / relationship type の意味と利用範囲を固定
- Standard API の lineage / context response を確認
- derived inquiry draft の入力契約を定義
- fixture の期待結果を固定

### Step 2: 系統樹の読み取り

- lineage view model
- lineage tree UI
- detail との相互遷移
- state / error handling

### Step 3: 文脈探索

- context query
- result view model
- context exploration UI
- comparison representation

### Step 4: 派生操作

- derived inquiry draft
- relation type selection
- review / approval
- publication
- lineage / provenance 反映

### Step 5: E2E と release validation

- Golden Path browser journey
- transport operational smoke
- replay verification
- accessibility / responsive checks
- docs / release notes / runbook 更新

## 9. Release Gate

v0.3.0 release candidate への移行には、次をすべて満たす必要があります。

- Issues #16, #17, #18 の必須完了条件を満たす
- release tracking issue #19 の横断チェックを完了する
- default CI が成功する
- Golden Path をブラウザ上で最後まで再現できる
- human review と publication guard が維持されている
- canonical identity / provenance / replay に既知回帰がない
- Nostr / Lingonberry operational smoke に既知回帰がない
- release notes / runbook / known limitations が更新されている

条件付きスコープの未完了は release blocker にしません。

---

## 10. 判断原則

v0.3.0 の設計・実装判断は、次の順で評価します。

1. 問いの循環と派生を強化するか
2. 利用者が問いの来歴と文脈を理解できるか
3. 人間の確認を迂回しないか
4. canonical semantics と identity を混同しないか
5. provenance と replayability を維持するか
6. transport 固有差分をUIへ過剰に露出しないか
7. 低資源端末・スマートフォンで利用可能か
8. テスト可能な明確な契約として実装できるか

---

## 11. 完了後の次段階

v0.3.0 の安定後に、Improvement Roadmap の Phase B 残項目を再評価します。

- 類似 inquiry 検索の高度化
- AI による relation 候補
- オフラインキュー
- identity state model の操作面
- 語彙の三層化

これらは v0.3.x または後続 minor release の候補とし、v0.3.0 の必須スコープとは分離します。
