## 1. 目的

このrunbookは、v0.3.0をrelease candidateとして判定し、タグ付け・公開へ進める際の再現可能な確認手順を定義します。

v0.3.0の中心的な確認対象は、雑草相Golden Pathを使った次の一連の流れです。

```text
inquiry detail
  -> lineage tree
  -> context exploration
  -> derived inquiry draft
  -> human review
  -> Canonical Event publication
  -> transport projection / re-ingest
  -> updated lineage tree
```

## 2. 前提

- Node.jsはCIと同じ対応バージョンを使う
- Corepackを利用可能にする
- repository rootで実行する
- working treeがcleanであることを確認する
- release対象commitが`main`に存在することを確認する

## 3. 必須Issue確認

次のIssueがclosedであることを確認します。

- #16 lineage tree
- #17 context exploration
- #18 reviewed derived inquiry publication

#19は本runbook、Release Notes、CI、横断E2Eの完了をもってclosedにします。

## 4. 依存関係とworkspace検証

```bash
corepack pnpm install --frozen-lockfile
corepack pnpm test
```

成功条件:

- installが終了コード0
- workspace testが終了コード0
- `@toitoi/frontend`からv0.3.0 cross-feature Golden Path testが実行される
- protocol、API、storage、transport、frontendの既存testsに回帰がない

## 5. v0.3.0横断Golden Path

対象:

```bash
corepack pnpm --filter @toitoi/frontend test
```

`test_v0_3_0_golden_path.js`は次を一続きで検証します。

1. Golden Path inquiry detailをStandard APIから取得する
2. 三段階lineageをfrontend modelとHTML rendererへ投影する
3. `climate_zone`によるcontext explorationを実行する
4. 関連inquiryを派生元として`translated_from` draftを作成する
5. 未承認draftのpublicationが拒否されることを確認する
6. human reviewで承認する
7. Canonical Eventを生成し、lineageとpublication provenanceを確認する
8. Nostrのrelation marker付き`e` tagへ投影する
9. transport eventを再取り込み・保存・replayする
10. 更新後lineage treeに派生inquiryが現れることを確認する
11. 派生inquiry detailからparentとNostr provenanceを確認する

## 6. Human review / publication guard

必須確認:

- `draft`、`in_review`、`rejected`は公開できない
- `approved`だけがCanonical Event化できる
- AI関与があってもhuman reviewを省略しない
- semantic relationは`lineage`へ記録する
- workflow provenanceは`meta.publication`へ記録し、`body.text`へ混入させない

## 7. Canonical semantics / provenance / replay

必須確認:

- frontendはStandard API canonical viewを使用する
- context similarityをcanonical identity mergeに使用しない
- lineage targetとrelation typeがre-ingest後も維持される
- provenance source protocolとsource IDをdetail viewから確認できる
- missing/cycle lineage referenceが画面全体を破壊しない

## 8. Primary transport regression

workspace testに含まれるdeterministic smokeで、少なくとも次を確認します。

- Nostr publish / retrieve / re-ingest
- Lingonberry publish / retrieve / re-ingest
- lineage / provenanceの既知回帰がない

外部relayまたはcarrierを利用するlive smokeは、環境変数で明示的に有効化する補助検証であり、default release gateには含めません。

## 9. Documentation review

次の整合性を確認します。

- `V0.3.0_RELEASE_PLAN.md`の必須スコープと実装が一致する
- `RELEASE_NOTES.md`がplannedではなくrelease candidateの実績を記載する
- `INQUIRY_DRAFT.md`が派生draft、review、publication provenanceを説明する
- frontend contract文書がlineageとcontext explorationを説明する
- known limitationsとnon-goalsが公開文書に残っている

## 10. Release candidate判定

次をすべて満たした場合、v0.3.0をrelease candidateとして扱えます。

- Issues #16、#17、#18がclosed
- default CIが成功
- cross-feature Golden Pathが成功
- publication guardが維持される
- canonical identity / provenance / replayに既知回帰がない
- Nostr / Lingonberry deterministic smokeに既知回帰がない
- release notesとrunbookが最終化されている

## 11. タグ付け前確認

```bash
git switch main
git pull --ff-only
git status --short
corepack pnpm install --frozen-lockfile
corepack pnpm test
```

working treeがcleanでtestsが成功したことを確認後、repositoryのrelease方針に従って`v0.3.0`タグとGitHub Releaseを作成します。

## 12. Known limitations

- framework-independent HTML renderer / contractが中心で、完成したSPA製品ではない
- 汎用graph editor、relation type自由編集、共同編集は含まない
- production-grade authentication / authorization / rate limitingは含まない
- embeddings必須の意味検索、graph inference、本格オフライン同期は含まない
- default transport smokeは決定的なローカル検証で、外部serviceの可用性を保証しない
- AIによる無人公開は行わない
