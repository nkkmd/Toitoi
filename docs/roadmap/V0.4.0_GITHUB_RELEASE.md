## Release body

Toitoi v0.4.0 introduces an asynchronous AI assistance layer in front of the human-reviewed Inquiry Draft workflow established in v0.3.0.

AI output does not modify Canonical Events or published inquiries directly. Summaries and tag candidates are stored as auditable annotations with model, prompt version, raw output, source event, and human review state. Only accepted annotations may support creation of an Inquiry Draft, and the existing `draft → in_review → approved` publication guard remains mandatory.

### Highlights

- Retry-bounded, duplicate-aware AI job queue separated from ingest
- Versioned AI annotation contract for summaries and tags
- Auditable provenance including model, prompt version, raw output, and review state
- Append-only JSONL persistence for job transitions and annotations
- Restart recovery with interrupted `processing` jobs explicitly requeued
- Provider-neutral worker boundary with a deterministic offline provider for CI
- Read-only Standard API inspection routes using `TOITOI_AI_STORAGE_DIR`
- Human-reviewed promotion of accepted annotations into derived Inquiry Drafts
- Preserved `derived_from` lineage and publication provenance after promotion
- Frontend review model that clearly distinguishes AI assistance from published inquiries
- Deterministic v0.4.0 Golden Path covering asynchronous inference through human-approved publication

### Standard API additions

When `TOITOI_AI_STORAGE_DIR` is configured, the following read-only endpoints are available:

- `GET /api/v1/ai/jobs`
- `GET /api/v1/ai/jobs/:id`
- `GET /api/v1/ai/annotations`
- `GET /api/v1/ai/annotations/:id`
- `GET /api/v1/ai/events/:eventId`

Available filters include `state`, `event_id`, `task`, and `review_state`.

The v0.4 API entrypoint also preserves protocol selection through `TOITOI_PROTOCOL`, including Nostr, ATProto, and Lingonberry configurations.

### Human-review boundary

The following rules are enforced:

1. Workers create annotations in the `unreviewed` state.
2. Unreviewed annotations cannot be promoted or published.
3. Human acceptance allows an annotation to be used as draft input only.
4. Promotion creates a derived Inquiry Draft with source lineage.
5. The draft must still pass through `in_review` and `approved`.
6. Only approved drafts may become published Canonical Events.

### Validation

The default GitHub Actions workflow validates:

- frozen-lockfile installation
- workspace-wide tests
- queue, annotation, worker, persistence, recovery, and promotion contracts
- AI inspection HTTP routes
- protocol selection through `TOITOI_PROTOCOL`
- AI provenance and review frontend behavior
- lineage preservation after AI-assisted promotion
- the complete v0.4.0 cross-feature Golden Path
- regressions across earlier frontend, API, protocol, and transport behavior

The final pre-merge validation completed successfully in CI run **#353**.

### Known limitations

- The deterministic provider is intended for contract and CI validation, not production inference.
- Production-grade distributed queues, authentication, authorization, and rate limiting are not included.
- The current AI API is read-only; annotation review and promotion mutation endpoints are not yet exposed over HTTP.
- The frontend remains a framework-independent view model and HTML renderer rather than a complete SPA.
- Generated content is not guaranteed to be agriculturally correct.
- Embeddings, vector databases, RAG, and semantic-identity merging remain future work.
- Fully autonomous publication of AI-generated inquiries is intentionally unsupported.

### Upgrade notes

No Canonical Event schema migration is required for this release.

To enable AI inspection in the API server, configure:

```bash
export TOITOI_AI_STORAGE_DIR=/path/to/ai-storage
```

For a single transport, protocol selection continues to use:

```bash
export TOITOI_PROTOCOL=nostr
# or: atproto / lingonberry
```
