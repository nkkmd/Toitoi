# Toitoi v0.5.0 — Practical Human-reviewed Inquiry Generation

Toitoi v0.5.0 extends the asynchronous AI annotation foundation introduced in v0.4.0 into practical, auditable inquiry generation.

A saved observation can now produce multiple structured inquiry candidates through either the deterministic CI provider or a local llama.cpp-compatible inference service. Every candidate remains an AI annotation until a human explicitly accepts or edits it. Promotion creates an Inquiry Draft with source lineage and AI provenance, and the existing independent publication review remains mandatory.

## Highlights

- Versioned `generate_inquiries` annotation contract for multiple inquiry candidates
- Structured retention of `inquiry`, `context`, `observation`, `relationship`, `uncertainty`, `tags`, and `source_refs`
- llama.cpp OpenAI-compatible production inference provider with timeout and bounded-output controls
- Provider-neutral worker boundary with deterministic, network-free CI validation
- Provenance covering model, model version, prompt version, raw output, creation time, and generation time
- Schema rejection for malformed JSON and malformed inquiry candidates before annotation persistence
- Append-only human review operations for accept, edit, and reject
- HTTP mutation routes for annotation review
- Promotion of accepted or human-edited candidates into valid Inquiry Drafts
- Preserved source observation lineage and AI annotation references
- Explicit separation between annotation review and publication approval
- A 4GB RAM-class low-resource execution profile
- Deterministic v0.5.0 Golden Path from observation through independently approved publication

## AI review API additions

The following mutation endpoints are available when the AI review service is configured:

- `POST /api/v1/ai/annotations/:id/accept`
- `POST /api/v1/ai/annotations/:id/edit`
- `POST /api/v1/ai/annotations/:id/reject`

Each mutation records the reviewer, review timestamp, review note, and resulting review state in append-only annotation storage.

## Human-review boundary

1. AI workers create inquiry-generation annotations in the `unreviewed` state.
2. Malformed model output is rejected before annotation persistence.
3. Unreviewed and rejected annotations cannot be promoted.
4. A human must explicitly accept or edit an annotation.
5. Promotion creates an Inquiry Draft and preserves source lineage and AI provenance.
6. Annotation acceptance does not approve publication.
7. The draft must still pass through `draft → in_review → approved`.
8. Only approved drafts may become published Canonical Events.

## Low-resource reference profile

```text
llama.cpp server: 1 process
AI worker concurrency: 1
context window: 2048 or lower
maximum generated tokens: 768
request timeout: 60 seconds
retry: bounded by the existing job queue
model class: 1.5B–1.7B, Q4 quantization
```

Inference remains outside the ingest and persistence critical path. A stopped or failing model service does not prevent observations or Canonical Events from being stored.

## Validation

The default GitHub Actions workflow validates:

- frozen-lockfile installation
- workspace-wide tests
- multi-candidate inquiry-generation schema validation
- deterministic inquiry generation without network access
- llama.cpp-compatible response parsing
- malformed-output rejection
- append-only accept, edit, and reject transitions
- repeated-review rejection
- unreviewed-promotion rejection
- valid Inquiry Draft promotion
- publication guard enforcement
- independent publication approval
- regressions across earlier API, frontend, protocol, storage, and transport behavior

Final validation succeeded in CI runs **#386**, **#392**, **#397**, **#404**, and **#410**.

## Known limitations

- Live llama.cpp validation is opt-in; default CI uses the deterministic provider.
- Model selection, installation, and model-file distribution remain operator responsibilities.
- The queue is local and single-process rather than distributed.
- Production authentication, authorization, and rate limiting are not included.
- The frontend remains a framework-independent boundary rather than a complete SPA / PWA.
- Offline observation storage and synchronization remain v0.6.0 scope.
- Generated inquiries are not guaranteed to be agriculturally correct.
- RAG, embeddings, vector search, and semantic-identity merging remain future work.
- Fully autonomous publication of AI-generated inquiries is intentionally unsupported.

## Upgrade notes

No Canonical Event schema migration is required for this release.

The AI annotation schema advances to support `generate_inquiries`, while existing `summarize` and `tag` tasks remain supported.

A llama.cpp-compatible endpoint can be configured for production inference. Default CI and offline contract validation do not require a running model service.
