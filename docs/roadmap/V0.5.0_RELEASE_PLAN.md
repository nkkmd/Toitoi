# Toitoi v0.5.0 Release Plan

**Status: implementation in progress** | **Baseline: v0.4.0** | **Started: 2026-07-19**

## Release intent

v0.5.0 extends the asynchronous annotation foundation with practical, auditable inquiry generation. A saved observation may produce multiple inquiry candidates through a local llama.cpp-compatible model, but every candidate remains an AI annotation until a human explicitly accepts or edits it and promotes it into an Inquiry Draft.

## Required scope

1. Production inference provider
   - connect to a llama.cpp OpenAI-compatible HTTP endpoint
   - retain the provider-neutral worker boundary and deterministic CI provider
   - enforce timeout and bounded output settings
2. Inquiry-generation annotation contract
   - support multiple candidates per observation
   - retain `inquiry`, `context`, `observation`, `relationship`, `uncertainty`, `tags`, and `source_refs`
   - reject malformed provider output before persistence
3. Provenance
   - retain model, model version, prompt version, raw output, creation time, and generation time
4. Human review mutation boundary
   - provide explicit accept, edit, and reject operations
   - record reviewer, review time, and review note
   - reject repeated or invalid state transitions
5. Promotion
   - allow only accepted or human-edited inquiry annotations to create an Inquiry Draft
   - preserve source observation lineage and AI annotation references
   - keep publication approval separate from annotation review
6. Low-resource profile
   - target one worker, one in-flight inference, small context, bounded tokens, timeout, and local-only service defaults
   - ensure inference failure never blocks observation ingest or canonical persistence
7. Golden Path
   - save observation
   - enqueue asynchronous `generate_inquiries` job
   - generate multiple candidates
   - accept or edit one candidate
   - promote it to an Inquiry Draft with lineage
   - reject publication until the existing human publication review approves the draft

## Acceptance criteria

- deterministic CI can reproduce inquiry generation without a network or model process
- a llama.cpp-compatible response can be parsed and validated
- malformed JSON or malformed candidate schema fails the AI job without modifying the source event
- each candidate traces back to source event IDs and raw model output
- accept, edit, and reject mutations are append-only and auditable
- accepted annotation state is not publication approval
- inference outage leaves ingest and persistence operational
- default workspace tests and the v0.5.0 Golden Path pass

## Explicitly out of scope

- unattended publication
- RAG, embeddings, or vector search
- semantic identity merge
- distributed queue infrastructure
- hosted external AI as a required dependency
- claims that model output is agriculturally correct

## Low-resource reference profile

```text
llama.cpp server: 1 process
AI worker concurrency: 1
context window: 2048 or lower
maximum generated tokens: 768
request timeout: 60 seconds
retry: bounded by the existing job queue
model recommendation: 1.5B-1.7B class, Q4 quantization
```

The deterministic provider remains the default validation provider. Live model validation is opt-in and must not be required by default CI.
