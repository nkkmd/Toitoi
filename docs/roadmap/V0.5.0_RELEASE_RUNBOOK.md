# Toitoi v0.5.0 Release Runbook

**Status: release candidate preparation** | **Last updated: 2026-07-19**

## Purpose

This runbook defines the verification and release procedure for v0.5.0. The release adds practical local-model inquiry generation while preserving the separation between AI annotation review and publication approval.

## Release gate

Run from the workspace root:

```bash
corepack pnpm install --frozen-lockfile
corepack pnpm test
```

The release candidate must pass the complete workspace suite, including:

- AI annotation and queue regression tests
- deterministic inquiry-generation tests
- llama.cpp-compatible response parsing tests
- malformed-output rejection tests
- annotation accept / edit / reject mutation tests
- candidate promotion tests
- v0.4.0 Golden Path regression
- v0.5.0 cross-feature Golden Path
- protocol, transport, storage, API, and frontend regressions

## Golden Path verification

The v0.5.0 Golden Path must demonstrate:

```text
saved observation
  → asynchronous generate_inquiries job
  → multiple validated inquiry candidates
  → explicit human accept or edit through the mutation API
  → selected candidate promoted to Inquiry Draft
  → draft publication rejected
  → independent human publication review
  → approved draft becomes publishable
```

Confirm that:

- source observation persistence does not depend on inference success
- raw model output and model/prompt provenance remain inspectable
- malformed candidate output is rejected before annotation persistence
- repeated review mutations are rejected
- rejected or unreviewed annotations cannot be promoted
- annotation acceptance does not approve publication
- the promoted candidate conforms to the existing Inquiry Draft contract

## Optional live llama.cpp check

Live model validation is optional and must not be required by default CI.

Recommended low-resource profile:

```text
llama.cpp processes: 1
AI worker concurrency: 1
context window: 2048 or lower
maximum generated tokens: 768
request timeout: 60 seconds
model class: 1.5B-1.7B, Q4 quantization
```

Verify the configured endpoint exposes an OpenAI-compatible `/v1/chat/completions` route. Use a non-sensitive observation fixture and inspect the resulting annotation before any promotion.

## Known limitations

- the completed user-facing SPA / PWA is scheduled for v0.6.0
- authentication, authorization, and rate limiting are not production-grade
- the queue is local and single-process, not distributed
- live model quality depends on the selected model and prompt
- AI-generated questions are suggestions, not validated agricultural conclusions
- RAG, embeddings, vector search, and semantic identity merge are out of scope
- unattended publication is prohibited

## Pull request readiness

Before marking the implementation PR ready for review:

1. confirm the final head commit has successful required CI
2. inspect the complete diff for unrelated changes
3. verify `V0.5.0_RELEASE_PLAN.md`, this runbook, and `RELEASE_NOTES.md` describe implemented behavior
4. confirm no API or protocol documentation claims unattended publication
5. confirm all new tests are included in the default workspace runner

## Merge and release

After required CI succeeds and review is complete:

1. squash-merge the implementation PR into `main`
2. run or confirm default CI on the resulting `main` commit
3. update release documentation from release candidate to released status
4. create tag `v0.5.0` from the verified `main` commit
5. create the GitHub Release using repository-maintained release content
6. verify the tag and release point to the same commit

Do not create the release tag from the feature branch or from a commit with pending or failing required checks.
