# Toitoi v0.6.0 Release Plan

**Status: active** | **Baseline: v0.5.0** | **Started: 2026-07-20** | **Last updated: 2026-07-20**

## Release intent

v0.6.0 integrates the existing Standard API, AI annotation review, Inquiry Draft workflow, publication guard, provenance, lineage, and context views into an installable mobile-first SPA/PWA. The release must let a user capture an observation while offline, review AI-generated inquiry candidates, complete human publication review, explicitly synchronize after connectivity returns, and inspect the published result without losing data or confusing local private material with public Canonical Events.

## Required scope

1. SPA application foundation
   - establish a production application entry point under `apps/frontend/`
   - retain reusable framework-independent view models and renderers where practical
   - define routing, application state, error boundaries, and API client boundaries
   - provide development, build, test, and preview scripts through the workspace
2. Observation capture
   - mobile-first observation input screen
   - local draft identifiers and timestamps
   - explicit handling of context, location, personal information, and private notes
   - local save without requiring network connectivity
3. AI inquiry-candidate review
   - display asynchronous job and annotation state
   - review multiple inquiry candidates
   - support accept, edit, and reject through the existing mutation API
   - preserve the distinction between AI annotation acceptance and publication approval
4. Inquiry Draft and publication review
   - edit the promoted Inquiry Draft
   - submit for human review and approve or reject through an explicit UI boundary
   - show publication guard failures as actionable states
   - provide a final pre-publication confirmation of public fields
5. Publication result and inspection
   - display publication success or failure
   - show inquiry detail, provenance, lineage, context, AI involvement, and review state
   - keep transport-specific representation secondary to the canonical user view
6. PWA installability
   - web app manifest
   - service worker and application-shell caching
   - installability checks in supported browsers
   - intentional update behavior that does not silently discard local work
7. Offline persistence and synchronization
   - IndexedDB-backed local observation and draft storage
   - offline publish queue with explicit states
   - user-triggered synchronization after connectivity recovery
   - retry and failure visibility
   - idempotency or duplicate-protection at the synchronization boundary
8. Privacy and publication boundary
   - raw local observations remain distinct from externally published Canonical Events
   - location, personal information, and non-public context require explicit review
   - no background publication without a human action
9. Mobile-first quality
   - primary Golden Path works on a smartphone-sized viewport
   - keyboard and screen-reader operability for core actions
   - resilient loading, empty, offline, partial-data, and error states
10. Golden Path E2E
    - capture an observation offline
    - persist it locally
    - reconnect and synchronize the observation
    - generate inquiry candidates asynchronously
    - accept or edit one candidate
    - promote it to an Inquiry Draft
    - complete independent human publication review
    - publish the approved event
    - inspect publication result, provenance, lineage, and context

## Implementation phases

### Phase 0: baseline integrity

- ensure all v0.5.0 Golden Path tests are included in the package test command
- confirm `corepack pnpm test` passes on the v0.5.0 baseline
- inventory Standard API endpoints needed by the SPA and identify missing mutation boundaries
- define browser support and local-development assumptions

### Phase 1: application shell and local observation storage

- select and configure the SPA toolchain without replacing stable domain modules unnecessarily
- add routing and an application shell
- add observation input and validation
- implement IndexedDB repositories for local observations, drafts, and synchronization records
- add online/offline status presentation

### Phase 2: connected review and publication flow

- implement the Standard API client
- add job and annotation status views
- add inquiry-candidate accept, edit, and reject operations
- add Inquiry Draft editing and publication review UI
- add pre-publication privacy confirmation and publication result views

### Phase 3: offline queue and explicit synchronization

- create a durable synchronization state machine
- queue network-dependent operations without publishing automatically
- add explicit synchronize, retry, and cancel actions
- handle application restart, duplicate attempts, partial failure, and server rejection
- verify that local raw observations are never projected as public events by accident

### Phase 4: PWA and mobile quality

- add manifest, icons, and service worker
- cache the application shell and safe read-only resources
- define update and cache invalidation behavior
- complete responsive layout, accessibility checks, and smartphone interaction refinement

### Phase 5: release validation

- add browser-level Golden Path E2E coverage
- add offline/online transition tests
- add synchronization retry and duplicate-protection tests
- document setup, operation, privacy boundaries, and recovery procedures
- synchronize README, implementation status, release notes, and GitHub Release content

## Acceptance criteria

- a user can install or launch the application as a PWA
- a smartphone-sized viewport can complete the primary workflow
- an observation created offline survives reload or application restart
- network loss does not erase a local observation or Inquiry Draft
- synchronization is explicit and its pending, running, failed, and completed states are visible
- retrying a synchronization operation does not create unintended duplicate public events
- AI inquiry candidates can be accepted, edited, or rejected through the UI
- accepting an AI annotation does not approve publication
- only an independently approved Inquiry Draft can be published
- the user sees and confirms public fields before publication
- local private fields are not included in the external Canonical Event unless explicitly selected
- publication results expose canonical identity, provenance, lineage, context, AI involvement, and human review state
- the complete UI Golden Path is reproducible in default CI without requiring a live model or external network
- existing v0.5.0 and earlier workspace tests remain green

## Architecture constraints

- the SPA consumes the Standard API and must not become dependent on Nostr, Lingonberry, or ATProto event shapes
- domain contracts remain in existing protocol, AI, storage, and API packages rather than being reimplemented in UI components
- IndexedDB records are local application state, not canonical public records
- service-worker behavior must not turn publication into an implicit background side effect
- synchronization transitions must be auditable and testable
- deterministic providers and fixtures remain the default CI path
- agricultural correctness remains a human judgment boundary; the UI must not present AI output as verified advice

## Initial data-state model

```text
local observation
  draft
  queued_for_sync
  syncing
  synced
  sync_failed

AI job
  queued
  running
  completed
  failed

AI annotation
  unreviewed
  accepted
  edited
  rejected

Inquiry Draft
  draft
  in_review
  approved
  rejected

publication
  not_requested
  ready
  publishing
  published
  failed
```

The UI may aggregate these states for presentation, but must not collapse the underlying review and publication boundaries.

## Explicitly out of scope

- unattended or background publication
- production multi-user authentication and authorization
- real-time collaborative editing
- push notifications as a required dependency
- semantic search, embeddings, vector search, or RAG
- the v0.7.0 semantic derivation authoring experience
- full transport administration UI
- native iOS or Android packages
- large-scale graph visualization
- destructive automatic conflict resolution

## Known baseline check

`apps/frontend/test_v0_5_0_golden_path.js` exists and is documented, but the current `apps/frontend/package.json` test script does not list it. Phase 0 must correct or intentionally explain this discrepancy before v0.6.0 feature work is considered reliable.

## First implementation slice

The first implementation PR should remain narrow and establish a trustworthy vertical foundation:

1. include the v0.5.0 Golden Path in the frontend package test command
2. add the SPA build/dev/test toolchain and application shell
3. implement the mobile observation form
4. persist observations to IndexedDB
5. show durable saved state and online/offline state
6. add tests proving an offline observation survives reload

This slice must not implement publication yet. It establishes the storage and UI boundary on which review, synchronization, and publication will depend.
