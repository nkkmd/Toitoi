# Toitoi v0.6.0 Release Plan

**Status: implementation** | **Baseline: v0.5.0** | **Started: 2026-07-20** | **Last updated: 2026-07-20**

## Release intent

v0.6.0 integrates the existing Standard API, AI annotation review, Inquiry Draft workflow, publication guard, provenance, lineage, and context views into an installable mobile-first SPA/PWA. A user can capture an observation while offline, explicitly synchronize it, review AI-generated inquiry candidates, complete independent human publication review, publish the approved Canonical Event through the existing storage and multi-transport boundaries, and inspect the result without confusing local private material with public records.

## Required scope

1. Mobile-first SPA/PWA application shell
2. IndexedDB-backed local observation persistence
3. Explicit offline publication queue with visible retry state
4. AI inquiry-candidate accept, edit, reject, and promotion operations
5. Inquiry Draft submission, approval, rejection, and publication UI
6. Human confirmation of location, personal information, and private context before synchronization
7. Standard API mutation boundary for observation, promotion, draft review, and publication
8. Canonical publication through append-only storage and multi-transport delivery
9. Publication result with lineage, AI provenance, human review, storage, and delivery state
10. Deterministic Golden Path and regression coverage

## Implemented architecture

```text
mobile SPA / PWA
  → IndexedDB local observation
  → explicit synchronization queue
  → Standard API workflow mutations
  → AI annotation human review
  → Inquiry Draft
  → independent publication review
  → approved Canonical Event
  → outbound multi-transport fan-out
  → append-only canonical storage
  → replay / Standard API discovery
```

The memory workflow service remains a deterministic test adapter. Production server assembly injects a canonical publisher that uses the selected protocol storage module and the existing outbound fan-out runtime.

## Implemented HTTP boundary

```text
POST /api/v1/observations
POST /api/v1/ai/annotations/:id/accept
POST /api/v1/ai/annotations/:id/edit
POST /api/v1/ai/annotations/:id/reject
POST /api/v1/ai/annotations/:id/promote
GET  /api/v1/inquiry-drafts/:id
POST /api/v1/inquiry-drafts/:id/submit
POST /api/v1/inquiry-drafts/:id/approve
POST /api/v1/inquiry-drafts/:id/reject
POST /api/v1/inquiry-drafts/:id/publish
GET  /api/v1/publications/:id
```

## Publication semantics

- raw observations remain local/private records and are not automatically public Canonical Events
- annotation acceptance permits promotion only; it is not publication approval
- only an independently approved Inquiry Draft may be published
- transport delivery failures do not erase the approved Canonical Event
- delivered, skipped, and quarantined transport results are retained in publication provenance
- the canonical publication is persisted through the selected protocol's append-only storage implementation
- replay remains the mechanism for rebuilding the Standard API index

## Acceptance criteria

- a smartphone-sized viewport can complete the primary workflow
- an observation created offline survives reload or application restart
- synchronization is explicit and pending, running, failed, and completed states are visible
- retrying synchronization does not create an unintended local duplicate
- AI inquiry candidates can be accepted, edited, or rejected through the UI
- accepting an AI annotation does not approve publication
- only an independently approved Inquiry Draft can be published
- sensitive and private fields require explicit confirmation
- publication results expose canonical identity, provenance, lineage, context, AI involvement, human review, storage record IDs, and transport delivery state
- canonical publication is written through the existing append-only storage boundary
- outbound publication uses the existing multi-transport runtime
- the complete Golden Path is reproducible in default CI without a live model or external service
- v0.5.0 and earlier workspace tests remain green

## Validation targets

- frontend offline Golden Path
- workflow HTTP mutation and publication guard test
- canonical publisher storage and transport adapter test
- API server request-body and mutation routing tests
- existing v0.3.0, v0.4.0, and v0.5.0 Golden Paths
- complete workspace CI

## Operational profile

```text
TOITOI_STORAGE_DIR=/path/to/storage
TOITOI_AI_STORAGE_DIR=/path/to/ai-storage
TOITOI_PROTOCOL=nostr
corepack pnpm --filter @toitoi/api start
```

`TOITOI_STORAGE_DIR` enables the workflow mutation and canonical publication runtime. `TOITOI_AI_STORAGE_DIR` enables AI annotation inspection and review. Live transport credentials remain optional; unavailable transports are reported as skipped or quarantined and do not bypass human publication review.

## Explicitly out of scope

- unattended or background publication
- production multi-user authentication and authorization
- real-time collaborative editing
- semantic search, embeddings, vector search, or RAG
- the v0.7.0 semantic derivation authoring experience
- native iOS or Android packages
- large-scale graph visualization
- destructive automatic conflict resolution
