# Toitoi v0.6.0 Release Runbook

**Status: release candidate** | **Target: v0.6.0** | **Last updated: 2026-07-20**

## 1. Release scope

v0.6.0 delivers the mobile-first SPA/PWA and offline Golden Path from local field observation to independently reviewed Canonical Event publication.

## 2. Required checks

```bash
corepack pnpm install --frozen-lockfile
corepack pnpm test
```

Confirm that default CI passes on the exact release commit and includes:

- v0.3.0-v0.5.0 Golden Path regressions
- v0.6.0 offline observation and retry Golden Path
- workflow HTTP mutation and publication guard test
- canonical storage and multi-transport publisher adapter test

## 3. Runtime smoke

```bash
export TOITOI_STORAGE_DIR=/tmp/toitoi-v0.6-storage
export TOITOI_AI_STORAGE_DIR=/tmp/toitoi-v0.6-ai
export TOITOI_PROTOCOL=nostr
corepack pnpm --filter @toitoi/api start
```

Verify:

```bash
curl http://127.0.0.1:3000/health
curl -X POST http://127.0.0.1:3000/api/v1/observations \
  -H 'content-type: application/json' \
  -d '{"text":"雨後に葉のしおれ方が変化した","language":"ja"}'
```

The observation response must remain private and local-only. Publication requires AI annotation review, Inquiry Draft promotion, independent Draft approval, and explicit publish.

## 4. PWA checks

Serve `apps/frontend/` over HTTP and confirm:

- manifest loads
- service worker installs
- observation can be saved while offline
- reload does not erase IndexedDB observations
- failed synchronization is visible and retryable
- sensitive fields require confirmation
- annotation acceptance is not displayed as publication approval
- only an approved Inquiry Draft enables publication

## 5. Storage and transport checks

- approved publication is appended through the selected protocol storage module
- publication provenance includes storage batch and record IDs
- outbound result distinguishes delivered, skipped, and quarantined transports
- unavailable external credentials do not cause unattended fallback publication
- replay can rebuild the Standard API index from storage

## 6. Documentation audit

Confirm these files describe implemented behavior:

- `docs/roadmap/V0.6.0_RELEASE_PLAN.md`
- `docs/roadmap/V0.6.0_RELEASE_RUNBOOK.md`
- `docs/roadmap/V0.6.0_GITHUB_RELEASE.md`
- `apps/frontend/README.md`
- `apps/api/README.md`
- root `README.md`

## 7. Merge and tag

1. mark the implementation PR ready
2. confirm required CI is successful
3. squash-merge into `main`
4. confirm CI succeeds on the resulting `main` commit
5. create annotated tag `v0.6.0` from that exact commit
6. publish the GitHub Release using `V0.6.0_GITHUB_RELEASE.md`
7. verify the release page and tag resolve to the same commit

## 8. Rollback boundary

Do not delete append-only publication records. If runtime behavior must be rolled back, revert application code and keep storage evidence. Transport failures remain auditable as quarantined delivery results.
