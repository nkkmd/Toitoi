# Toitoi v0.7.0

v0.7.0 makes inquiry derivation and lineage a first-class user workflow.

A user can start from an existing published inquiry, choose and explain a semantic relation, create a derived Inquiry Draft, complete independent human review, publish a new Canonical Event, and inspect the resulting lineage, provenance, AI involvement, and review state.

## Highlights

- eight first-class semantic relation types:
  - `derived_from`
  - `translated_from`
  - `observed_alongside`
  - `contrasts_with`
  - `synthesizes`
  - `reframes`
  - `annotates`
  - `revises`
- relation-specific input guidance and deterministic validation
- `POST /api/v1/inquiries/:id/derive`
- mobile-first derivation authoring UI
- explicit human confirmation of the selected semantic relation
- separate recording of AI-suggested and human-confirmed relation state
- preservation of multiple source inquiry IDs for synthesis
- integrated lineage, context, identity, provenance, AI, and human-review presentation
- transport projection, re-ingest, and replay fixture for lineage reconstruction
- continued enforcement of the existing Inquiry Draft publication guard

## Semantic safety boundary

v0.7.0 deliberately separates three concepts:

1. **Canonical identity** — whether records represent the same canonical event
2. **Semantic relation** — how one inquiry derives from or relates to another
3. **Context similarity** — whether inquiries share contextual characteristics

Context similarity and semantic relation do not cause automatic identity merge. AI may propose a relation, but a human must explicitly confirm the relation before a derived Inquiry Draft is created. Publication still requires an independent human review.

## Golden Path

```text
published inquiry
  → choose a semantic relation
  → complete relation-specific fields
  → explicitly confirm the relation
  → create a derived Inquiry Draft
  → submit for review
  → independent human approval
  → publish a distinct Canonical Event
  → inspect lineage / context / provenance
  → project to transport
  → re-ingest and replay
  → recover equivalent lineage semantics
```

## Validation

The default workspace test suite covers:

- all eight relation types
- relation-specific validation failures
- derivation workflow HTTP mutation
- human-confirmed versus AI-suggested relation separation
- mobile derivation form state
- integrated genealogy presentation
- transport round trip and storage replay
- v0.6.0 and earlier regression suites

## Upgrade notes

No schema migration or new service is required. v0.7.0 extends the existing protocol, workflow mutation, Standard API, SPA/PWA, canonical storage, and multi-transport boundaries.

Existing derived inquiry code that compares the full `DERIVATION_TYPES` array should update its expected list to include `observed_alongside` and `contrasts_with` in the canonical order.

## Known limitations

- production authentication, authorization, rate limiting, and multi-user tenancy remain out of scope
- semantic search, embeddings, vector search, and RAG are not included
- large-scale graph visualization and graph inference are not included
- no automatic semantic identity merge is performed
- AI-selected relations without human confirmation are rejected
- generated or derived inquiries are not guaranteed to be agriculturally correct
- live external transport checks remain operationally optional; deterministic fixtures run in default CI
