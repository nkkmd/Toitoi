# Toitoi v0.7.0 Release Plan

**Status: active** | **Baseline: v0.6.0** | **Started: 2026-07-21** | **Last updated: 2026-07-21**

## Release intent

v0.7.0 turns inquiry lineage from a read-only inspection capability into a first-class authoring workflow. A user can derive a new Inquiry Draft from an existing published inquiry, explicitly select and explain the semantic relation, complete human review, publish through the existing Canonical Event and multi-transport boundaries, and trace the result back through provenance and lineage.

## Required scope

1. First-class semantic relation vocabulary:
   - `derived_from`
   - `translated_from`
   - `observed_alongside`
   - `contrasts_with`
   - `synthesizes`
   - `reframes`
   - `annotates`
   - `revises`
2. Relation-specific authoring guidance and validation.
3. Creation of a derived Inquiry Draft from an existing inquiry.
4. Explicit separation of human-confirmed relations from AI-suggested relations.
5. Standard API workflow mutation for derivation.
6. SPA/PWA API client support for derivation.
7. Publication metadata containing source inquiry, semantic relation, author, AI involvement, and human review.
8. Deterministic tests covering all relation types and the derivation-to-publication Golden Path.
9. Lineage reconstruction after canonical storage replay and transport round trip.
10. Documentation for operators and frontend integrators.

## Semantic invariants

- Canonical identity, semantic relation, and context similarity remain separate concepts.
- Similarity never causes automatic identity merge.
- A relation is not authoritative until a human explicitly confirms it.
- AI may suggest a relation, but the selected relation and confirmation state are recorded independently.
- A derived Inquiry Draft follows the existing `draft → in_review → approved → published` boundary.
- Publication remains impossible without independent human approval.
- Transport delivery failure does not erase the approved Canonical Event.

## Relation validation

| Relation | Required authoring signal |
|---|---|
| `derived_from` | General derivation note is optional |
| `translated_from` | Target language must differ from the source language |
| `observed_alongside` | Context or observation note is required |
| `contrasts_with` | Contrast rationale is required |
| `synthesizes` | At least two source inquiry IDs are required |
| `reframes` | Reframing rationale is required |
| `annotates` | Annotation text or rationale is required |
| `revises` | Revision summary is required |

## Golden Path

```text
published inquiry
  → choose derive action
  → choose semantic relation
  → complete relation-specific fields
  → create derived Inquiry Draft
  → submit for review
  → independent human approval
  → publish Canonical Event
  → inspect provenance and lineage
  → replay storage / transport projection
  → recover the same lineage semantics
```

## Acceptance criteria

- [ ] All eight relation types are accepted by the protocol contract.
- [ ] Invalid relation-specific input is rejected deterministically.
- [ ] A user can create a derived Inquiry Draft from an existing inquiry through the workflow HTTP boundary.
- [ ] Human-confirmed and AI-suggested relation state remain distinguishable.
- [ ] Existing Draft review and publication guard remain mandatory.
- [ ] Published events preserve source inquiry IDs and semantic relation metadata.
- [ ] Lineage tree can trace a published derivation to its source inquiry and original observation.
- [ ] Transport replay restores equivalent lineage.
- [ ] v0.6.0 and earlier tests remain green.
- [ ] Mobile-first frontend can invoke the derivation workflow.

## Explicitly out of scope

- semantic search, embeddings, vector search, or RAG
- automatic identity merge
- graph inference
- large-scale graph visualization
- real-time collaborative editing
- production multi-user authorization
- AI-selected relations without human confirmation
- new transport implementations

## Implementation sequence

1. Protocol relation contract and validation.
2. Workflow service and HTTP mutation boundary.
3. Frontend API and authoring state.
4. Lineage, context, and provenance presentation integration.
5. Replay and transport round-trip fixtures.
6. Golden Path and regression validation.
7. Release documentation and release administration.
