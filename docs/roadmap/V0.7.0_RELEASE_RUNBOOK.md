# Toitoi v0.7.0 Release Runbook

**Status: release-ready** | **Baseline: v0.6.0** | **Last updated: 2026-07-21**

## Purpose

This runbook validates and publishes v0.7.0, which makes semantic inquiry derivation a first-class human-reviewed workflow while preserving canonical identity, provenance, lineage, replay, and transport independence.

## Release invariants

- Canonical identity, semantic relation, and context similarity remain visually and semantically separate.
- Similarity never causes automatic identity merge.
- AI may suggest a relation, but a human must explicitly select and confirm the relation.
- A derived Inquiry Draft must pass the existing `draft → in_review → approved` workflow.
- Only an independently approved Draft may be published.
- Transport delivery failure must not erase the approved Canonical Event.

## Required environment

- Node.js supported by the repository CI configuration
- Corepack
- pnpm 11.3.0 through the workspace `packageManager` declaration
- clean checkout of the verified v0.7.0 release commit

## 1. Install dependencies

```bash
corepack enable
corepack pnpm install --frozen-lockfile
```

## 2. Run complete deterministic validation

```bash
corepack pnpm test
```

The default suite must include and pass:

- protocol relation contract and legacy derived-inquiry regression
- workflow derivation HTTP mutation test
- mobile derivation form-model test
- integrated genealogy view test
- v0.7.0 transport replay round-trip fixture
- all v0.6.0 and earlier Golden Paths

## 3. Verify relation contract

Confirm that the protocol exports exactly these first-class relation types:

```text
derived_from
translated_from
observed_alongside
contrasts_with
synthesizes
reframes
annotates
revises
```

Verify deterministic rejection of missing relation-specific fields and confirm that `relationConfirmedByHuman` must be `true` before a derived Draft can be created.

## 4. Verify mobile derivation workflow

Start the API using the existing storage and transport configuration:

```bash
TOITOI_STORAGE_DIR=/path/to/storage \
TOITOI_AI_STORAGE_DIR=/path/to/ai-storage \
TOITOI_PROTOCOL=nostr \
corepack pnpm --filter @toitoi/api start
```

Serve `apps/frontend/` from a static HTTP server on the same origin or through a development proxy.

Verify on a smartphone-sized viewport:

1. enter an existing published inquiry ID
2. select a semantic relation
3. complete relation-specific fields
4. enter the derived inquiry text
5. explicitly confirm the relation as a human decision
6. create the Inquiry Draft
7. submit, approve, and publish using the existing review workflow
8. inspect lineage, provenance, AI involvement, and human review

## 5. Verify semantic separation

For a derived inquiry, verify that the UI and API do not conflate:

- the derived Canonical Event ID
- the semantic relation and source inquiry IDs
- shared or similar context values

The derived event must retain a distinct Canonical Event ID even when its contexts are similar to the source.

## 6. Verify transport replay

Run the v0.7.0 transport replay fixture through the default test suite. Confirm that:

- the semantic relation is projected to the transport representation
- the projected event can be ingested again
- append-only storage replay rebuilds the lineage edge
- the derived event remains distinct from the source event
- context data remains available after replay

## 7. Documentation gate

Confirm that the following describe the same scope and limitations:

- `V0.7.0_RELEASE_PLAN.md`
- `V0.7.0_RELEASE_RUNBOOK.md`
- `V0.7.0_GITHUB_RELEASE.md`
- `RELEASE_NOTES.md`
- `FOUNDATION_IMPLEMENTATION_STATUS.md`
- root, API, and frontend README files

## 8. Merge gate

Before merging the implementation PR:

- all GitHub Actions checks are successful
- no unresolved review thread remains
- the PR is mergeable against `main`
- the release plan acceptance criteria are checked
- release documentation contains no incorrect final commit SHA

## 9. Release administration

After the implementation PR is merged:

1. record the merged implementation SHA in the release plan and release notes
2. run the complete workspace test suite on `main`
3. create annotated tag `v0.7.0` at the verified release commit
4. publish the GitHub Release using `V0.7.0_GITHUB_RELEASE.md`
5. verify that the public tag and release page target the intended commit
6. update release records from `release-ready` to `released`

## Rollback guidance

Do not delete approved Canonical Events to hide transport failure. If release validation fails:

- do not create or move the `v0.7.0` tag
- revert the release candidate through a normal reviewed commit
- preserve append-only storage and provenance records
- document the failing acceptance criterion and rerun the deterministic fixture after correction
