# Toitoi v0.8.0 Release Plan

**Status: active** | **Baseline: v0.7.0** | **Started: 2026-07-22** | **Last updated: 2026-07-22**

## Release intent

v0.8.0 makes accumulated inquiries discoverable and reusable across words, contexts, regions, semantic relations, provenance, and review state. Search is introduced as a rebuildable derived projection over Canonical Events. It does not become a new source of truth and it never changes canonical identity.

## Required scope

1. SQLite FTS5 search across inquiry text, observation text, accepted summaries, and tags.
2. Structured filters for context, semantic relation, transport, provenance, and review state.
3. Context exploration for region, climate, soil, crop, and season.
4. Explicit separation of exact identity, explicit semantic relation, and related candidate.
5. Core Vocabulary, Domain Vocabulary, and Local Vocabulary contracts.
6. Mapping and translation claims that connect local terms to shared concepts without erasing the local label.
7. Search-result navigation to inquiry detail, provenance, lineage, context, and the existing derivation workflow.
8. A fixed reference dataset and deterministic search-quality regression tests.
9. Replay-safe search index rebuilding from canonical storage.
10. Operator and frontend documentation for search and vocabulary behavior.

## Implementation order

```text
FTS5 projection
  → structured filters
  → context / relation exploration
  → vocabulary registry and mapping claims
  → related-candidate ranking
  → search and reuse UI
  → reference dataset and regression evaluation
  → optional embedding boundary
```

Optional embeddings may only be introduced after the lexical and structured baseline is complete. They may rank related candidates but may not merge identity.

## Architectural boundaries

- Canonical storage remains the source of truth.
- The FTS5 database is derived data and must be rebuildable.
- Search failure must not block ingest, persistence, review, or publication.
- Exact identity requires the existing canonical identity or an explicit identity claim.
- Semantic relations remain explicit lineage edges.
- Similarity produces related candidates only.
- Vocabulary mappings are claims with provenance, not silent normalization.
- Local vocabulary labels remain visible in search results and detail views.
- Existing `draft → in_review → approved → published` boundaries remain mandatory.
- No new transport, persistence authority, or publication path is introduced.

## Search document projection

Each indexable Canonical Event is projected into a search document containing:

- canonical event ID
- event type
- primary text
- observation text when present
- accepted summary text when present
- tags and labels
- context keys and values
- relation types and targets
- transport and provenance identifiers
- review state
- creation timestamp

The projection stores the canonical ID as an unindexed lookup key. Search results resolve back to the current canonical read model before presentation.

## Vocabulary model

### Core Vocabulary

Stable cross-domain concepts required by the Toitoi contract, including semantic relation types and common context dimensions.

### Domain Vocabulary

Concepts maintained for a field such as agroecology, soil science, crop production, or local ecology.

### Local Vocabulary

Community, regional, dialectal, and practice-specific terms. Local terms are not rewritten into shared terms. They are connected using explicit mapping or translation claims.

### Mapping claim invariants

- source term and target term remain distinct identifiers
- mapping type is explicit, such as `equivalent_to`, `broader_than`, `narrower_than`, `related_to`, or `translated_as`
- author, provenance, review state, and creation time are retained
- mappings do not create Canonical Event identity
- disputed or alternative mappings may coexist

## API target

```text
GET /api/v1/search
GET /api/v1/search/contexts
GET /api/v1/search/related/:id
GET /api/v1/vocabularies
GET /api/v1/vocabularies/:scope/terms
GET /api/v1/vocabulary-mappings
```

The existing inquiry query endpoints remain compatible during v0.8.0. The new search boundary provides ranked results, structured facets, and explicit match classification.

## Result classification

Every search result must expose one of the following classifications:

- `exact_identity`: resolved through canonical ID or explicit identity claim
- `explicit_relation`: connected by a stored semantic relation
- `related_candidate`: ranked by lexical, contextual, or optional embedding similarity

A result may expose supporting signals, but classification must not be inferred as identity from similarity.

## Acceptance criteria

- [ ] FTS5 is available through a rebuildable search projection.
- [ ] Inquiry, observation, accepted summary, tags, and labels are searchable.
- [ ] Context, relation, transport, provenance, and review-state filters compose deterministically.
- [ ] Region, climate, soil, crop, and season facets can be explored.
- [ ] Search results link to canonical detail, provenance, lineage, and derivation.
- [ ] Core, Domain, and Local Vocabulary scopes are represented explicitly.
- [ ] Mapping and translation claims retain local terms and provenance.
- [ ] Exact identity and related candidates are distinguishable in API and UI.
- [ ] Replaying the same canonical fixture rebuilds equivalent search results.
- [ ] A fixed reference dataset detects ranking and filtering regressions.
- [ ] Search index failure does not stop canonical ingest or publication.
- [ ] v0.7.0 and earlier tests remain green.

## Explicitly out of scope

- automatic identity merge from embeddings or lexical similarity
- production RAG
- graph inference
- migration to a distributed search engine
- new transport implementations
- production authentication, authorization, or rate limiting
- autonomous AI publication
