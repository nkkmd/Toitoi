# Toitoi v0.9.0 GitHub Release

## Release metadata

- tag: `v0.9.0`
- title: `Toitoi v0.9.0 — Collaborative Operation, Safety, and Conformance`
- target: to be recorded after merge

## Release body

Toitoi v0.9.0 raises the v0.8.0 search, vocabulary, lineage, offline workflow, and human-reviewed inquiry foundation into a small-group operational reference system with explicit authority, durable recovery, stable API failure behavior, and externally runnable compatibility checks.

### Highlights

- actor identity and explicit reader, contributor, reviewer, publisher, moderator, and operator roles
- stable JSON error envelopes with request IDs
- mutation idempotency and conflict detection
- deterministic rate limiting
- append-only hash-chained audit records
- liveness and dependency readiness endpoints
- authenticated server mode while preserving deterministic development fixtures
- durable JSONL job queue with retry, dead-letter, restart recovery, and graceful shutdown behavior
- backup manifests with path, size, and SHA-256 verification
- migration registry with plan, dry-run, target version, and apply operations
- external `toitoi-conformance` CLI
- Canonical Event, provenance/rawRef, Canonical ID, replay-equivalence, and semantic round-trip checks
- fixed Nostr, Lingonberry, and ATProto conformance fixture
- governance, moderation, information-protection, correction, withdrawal, and deletion-request procedures

### Operational boundary

Production-style authority enforcement is enabled through documented configuration. Actor identity, authentication identity, transport identity, and Canonical Event identity remain separate. Annotation acceptance, relation confirmation, publication approval, moderation, and operator authority remain distinct auditable decisions.

Canonical storage remains the source of truth. Queue state, audit projections, metrics, and search indexes do not replace Canonical storage. Derived indexes remain rebuildable.

### Conformance

External implementations can run:

```bash
node packages/conformance/cli.js \
  --input fixtures/conformance/v0.9.0-transport-round-trips.json \
  --pretty
```

The suite reports machine-readable results and exits non-zero on conformance failure or invalid CLI input.

### Safety boundaries

- no automatic identity merge
- no autonomous AI publication or moderation
- no silent historical rewrite for moderation, correction, withdrawal, or deletion requests
- no new transport implementation
- no distributed consensus or multi-primary storage
- transport failure does not invalidate Canonical persistence

### Validation

The release candidate must pass:

- complete workspace CI
- v0.9.0 authority, idempotency, rate-limit, audit, health, queue, recovery, migration, and Conformance tests
- Nostr, Lingonberry, and ATProto semantic fixture checks
- live-ingest/replay equivalence
- v0.8.0 and earlier regression tests

### Upgrade notes

Existing development fixtures continue to operate without actor headers. Deployments enabling authenticated operation must provide actor identity and role headers through a trusted authentication boundary and must provide an `Idempotency-Key` for mutation requests.

Before production use, operators must configure durable storage locations, backup retention, migration state, rate limits, actor provisioning, audit retention, and information-protection policy.

### Known limitations

- the included deterministic actor-header provider is an operational boundary and testable reference, not OAuth/OIDC certification
- queue and recovery tooling target a documented single-node durability profile
- Conformance fixtures validate semantic contracts and do not require live public transport credentials
- legal and regulatory compliance remains deployment- and jurisdiction-specific
- large-scale federation, distributed queues, and distributed search remain outside v0.9.0
