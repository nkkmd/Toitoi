# Toitoi v0.9.0 Release Plan

**Status: implementation** | **Baseline: v0.8.0** | **Started: 2026-07-22** | **Last updated: 2026-07-22**

## Release intent

v0.9.0 raises Toitoi from a single-developer reference environment to a small-group operational system with explicit authority boundaries, durable recovery procedures, stable API contracts, externally runnable conformance tests, and documented governance.

The release does not replace Canonical storage, introduce a new transport, or make AI output authoritative. Existing review, publication, provenance, lineage, replay, and no-identity-merge boundaries remain mandatory.

## Required scope

### API and authority

- actor identity separated from Canonical Event identity and transport identity
- authentication boundary with a deterministic development provider
- role-based authorization for read, contribute, review, publish, moderate, and operate actions
- append-only audit records for security-sensitive mutations
- rate-limit policy and deterministic limiter
- cursor-compatible pagination contract
- idempotency-key contract for retried mutations
- stable JSON error envelope

### Durable single-node operation

- explicit single-node durability profile
- durable job-state contract and dead-letter handling
- graceful shutdown hooks
- backup and restore manifest
- replay and derived-index rebuild runbook
- schema migration registry and dry-run support
- liveness, readiness, and dependency health reports
- structured logs and minimal metrics
- per-transport delivery status preservation

### Conformance Suite

- Canonical Event JSON Schema validation
- required fields and schema-version checks
- Canonical ID preservation checks
- provenance and rawRef responsibility checks
- unknown-field forward-compatibility checks
- migration fixture checks
- Nostr, Lingonberry, and ATProto semantic round-trip adapters
- live-ingest and replay equivalence checks
- shared valid, invalid, migration, transport, and replay fixtures
- machine-readable conformance report

### Governance and protection

- Canonical Event change procedure
- schema versioning and compatibility policy
- TIP proposal, discussion, and adoption process
- vocabulary change procedure
- deprecated-field policy
- moderation principles
- protection of local, traditional, and community knowledge
- location, personal, and confidential information policy
- correction, withdrawal, deletion-request, and append-only-storage policy

## Architectural boundaries

1. Canonical storage remains the source of truth.
2. Search indexes, metrics, queue indexes, and health projections remain rebuildable derived state.
3. Authentication identity, actor identity, transport identity, and Canonical Event identity remain distinct.
4. Annotation acceptance, relation confirmation, publication approval, moderation, and operator authority remain separate decisions.
5. Moderation and withdrawal create auditable state transitions; they do not silently rewrite historical records.
6. A transport outage may affect delivery status but must not invalidate Canonical persistence.
7. Idempotency prevents duplicated effects; it does not suppress distinct reviewed actions.
8. Conformance tests validate observable contracts rather than private implementation details.

## Role model

| Role | Read | Contribute | Review | Publish | Moderate | Operate |
|---|---:|---:|---:|---:|---:|---:|
| `reader` | yes | no | no | no | no | no |
| `contributor` | yes | yes | no | no | no | no |
| `reviewer` | yes | yes | yes | no | no | no |
| `publisher` | yes | yes | yes | yes | no | no |
| `moderator` | yes | yes | yes | no | yes | no |
| `operator` | yes | yes | yes | yes | yes | yes |

A deployment may grant multiple roles to one actor. Publication approval and moderation remain independently auditable even when performed by the same operator in a small deployment.

## Stable error contract

```json
{
  "error": {
    "code": "forbidden",
    "message": "The actor lacks the required capability.",
    "requestId": "req_...",
    "details": {}
  }
}
```

Error codes are stable machine contracts. Human-readable messages may improve without changing the code.

## Pagination contract

List endpoints retain `limit` and `offset` compatibility while adding an opaque `cursor` and `nextCursor`. New clients should prefer cursor pagination where deterministic ordering is available.

## Idempotency contract

Mutation endpoints accept `Idempotency-Key`. A repeated key with the same actor, operation, and canonical request digest returns the recorded result. Reusing a key with a different digest returns `idempotency_conflict`.

## Implementation sequence

```text
stable errors / pagination / idempotency
  → actor identity / authentication / authorization
  → audit log / rate limiting
  → durable queue / dead-letter / graceful shutdown
  → backup / restore / migration / health / observability
  → Conformance Suite
  → governance and protection documents
  → cross-feature E2E and release validation
```

## Acceptance criteria

- [ ] multiple actors can be operated with explicit capability separation
- [ ] unauthorized review, publication, moderation, and operator actions fail with stable errors
- [ ] security-sensitive actions produce append-only audit records
- [ ] mutation retries are idempotent and conflicts are explicit
- [ ] rate limiting is deterministic and does not corrupt persisted work
- [ ] storage, queue state, and derived indexes can be restored after a simulated failure
- [ ] graceful shutdown stops new work and preserves in-flight state
- [ ] migrations support plan, dry-run, apply, and recorded version state
- [ ] health endpoints distinguish process liveness from dependency readiness
- [ ] logs are structured and metrics expose queue, error, and transport-delivery state
- [ ] external implementations can run the Conformance Suite without importing private application modules
- [ ] semantic round trips preserve meaning across Nostr, Lingonberry, and ATProto fixtures
- [ ] live ingest and replay produce equivalent Canonical and derived results
- [ ] governance, moderation, privacy, withdrawal, and compatibility responsibilities are documented
- [ ] v0.8.0 and earlier regressions remain green

## Explicitly out of scope

- internet-scale identity federation
- OAuth/OIDC provider certification
- distributed consensus or multi-primary storage
- a distributed queue or distributed search engine
- automatic identity merge
- autonomous AI publication or moderation
- legal guarantees for every jurisdiction
- new transport implementations
