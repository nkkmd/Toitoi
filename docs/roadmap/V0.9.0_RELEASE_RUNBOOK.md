# Toitoi v0.9.0 Release Runbook

**Status: implementation validation** | **Baseline: v0.8.0** | **Last updated: 2026-07-22**

## Purpose

This runbook validates the v0.9.0 collaborative-operation, safety, durability, and external-compatibility release scope without weakening Canonical storage, human review, provenance, lineage, replay, or no-identity-merge boundaries.

## Preconditions

- Node.js 24
- Corepack
- pnpm 11.3.0
- clean checkout of `agent/v0.9.0-operations-conformance`
- no production credentials are required for deterministic validation

## 1. Install and run the complete workspace suite

```bash
corepack enable
corepack prepare pnpm@11.3.0 --activate
corepack pnpm install --frozen-lockfile
corepack pnpm test
```

The suite must include:

- authority and role separation
- stable error envelopes
- idempotency replay and conflict handling
- deterministic rate limiting
- hash-chained audit verification
- liveness and readiness reports
- API operations-boundary integration
- durable JSONL queue recovery and dead-letter handling
- graceful shutdown behavior
- backup manifest and restore verification
- migration planning, dry-run, and apply behavior
- Conformance Suite library and CLI tests
- Nostr, Lingonberry, and ATProto semantic round-trip fixture
- live-ingest and replay equivalence
- v0.8.0 and earlier regressions

## 2. Validate development compatibility mode

Run the existing Standard API, search, vocabulary, related-inquiry, AI, and workflow tests without actor headers.

Expected behavior:

- direct `createToitoiApiService()` use retains its existing synchronous contract where the underlying route is synchronous
- existing read fixtures remain available
- existing mutation fixtures do not require a new idempotency header
- no existing publication or review boundary is bypassed

## 3. Validate authenticated operation mode

Start the API with:

```bash
TOITOI_AUTH_REQUIRED=true \
TOITOI_STORAGE_DIR=/path/to/storage \
TOITOI_AI_STORAGE_DIR=/path/to/ai-storage \
corepack pnpm --filter @toitoi/api start
```

Check:

```text
GET /health/live
GET /health/ready
GET /api/v1/audit
```

Authenticated requests use:

```text
X-Toitoi-Actor-Id: actor-id
X-Toitoi-Roles: contributor,reviewer
Idempotency-Key: unique-mutation-key
```

Verify:

- missing actor identity returns `authentication_required`
- insufficient authority returns `forbidden`
- mutation without `Idempotency-Key` returns `idempotency_key_required`
- repeated identical mutation is replayed
- reused key with changed request returns `idempotency_conflict`
- rate-limit rejection does not mutate persisted state
- operator-only audit access is enforced

## 4. Validate durable queue recovery

Use `JsonlDurableQueue` with a temporary directory.

1. enqueue pending work
2. claim one item
3. restart the queue instance
4. confirm unfinished work is recoverable
5. fail work through the configured retry limit
6. confirm the final state is `dead_letter`
7. begin shutdown and confirm new claims are rejected

Queue state is operational state. Canonical Events remain in Canonical storage and must not be reconstructed solely from queue records.

## 5. Validate backup and restore

Build a backup manifest covering at minimum:

- Canonical storage
- raw transport storage
- AI annotation storage
- durable queue state
- audit records
- migration state
- operator configuration required for restoration

Verify every manifest entry by relative path, size, and SHA-256. Modify one copied file and confirm verification fails.

After restoration:

1. validate Canonical storage
2. replay transport records where required
3. restore pending queue state
4. rebuild search and other derived indexes
5. verify readiness
6. run the reference search and workflow fixtures

## 6. Validate migrations

For every registered migration:

```text
plan → dry-run → backup → apply → record version → health check → focused regression
```

Confirm that:

- migration IDs are unique
- ordering is deterministic
- target-version planning stops at the requested version
- dry-run causes no mutation
- apply records the resulting version
- rollback is performed by restoring the verified backup unless the migration explicitly documents a safe reverse operation

## 7. Run the external Conformance CLI

```bash
node packages/conformance/cli.js \
  --input fixtures/conformance/v0.9.0-transport-round-trips.json \
  --pretty
```

Expected result:

- exit status `0`
- `conformanceVersion` is `0.9.0`
- Nostr round trip passes
- Lingonberry round trip passes
- ATProto round trip passes
- replay equivalence passes
- no fixture causes Canonical identity merge

An external implementation may produce the same input contract without importing Toitoi application modules.

## 8. Governance and protection review

Confirm publication of:

- TIP and Canonical Event change procedure
- schema compatibility and deprecation policy
- vocabulary change procedure
- moderation principles
- local, traditional, and community knowledge protections
- location, personal, and confidential information handling
- correction, withdrawal, and deletion-request handling
- append-only storage responsibility boundaries

## 9. Merge gate

Before merging PR #37:

- complete workspace CI is green on the final head
- all v0.9.0 focused tests are green
- v0.8.0 and earlier regressions are green
- no unresolved review thread remains
- PR is mergeable against `main`
- release-facing documentation matches implementation
- known limitations are explicit
- production authentication remains opt-in through documented configuration until deployment credentials and policy are configured

## 10. Post-merge release sequence

1. validate merged `main`
2. synchronize README, API README, implementation status, and release notes
3. prepare the GitHub Release body
4. record the merged commit and final CI run
5. create annotated tag `v0.9.0`
6. publish `Toitoi v0.9.0 — Collaborative Operation, Safety, and Conformance`
7. verify the public tag, release page, and documentation links

## Rollback guidance

- stop new work and enter maintenance mode
- preserve audit, queue, and delivery records
- restore the latest verified backup
- replay append-only storage
- rebuild derived indexes
- verify readiness and Conformance Suite results
- document the incident and any withdrawn release artifact

Never rewrite published history silently. Corrections and withdrawals remain auditable state transitions.
