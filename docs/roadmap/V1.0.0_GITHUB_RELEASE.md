# Toitoi v1.0.0 GitHub Release Draft

> Status: draft — do not publish until the release runbook is complete and the implementation PR has been merged and revalidated on `main`.

## Toitoi v1.0.0 — Minimum Reference Implementation

Toitoi v1.0.0 completes the first reproducible minimum reference implementation of a protocol-independent knowledge space in which inquiries grow from field observations, pass through explicit human review, are published as Canonical Events, are rediscovered and derived in other contexts, and retain auditable provenance and lineage.

The release is centered on one deterministic reference scenario:

> An observation that only the east side of a field has different weed species produces human-reviewable inquiry candidates, becomes a published Canonical Event, is connected to a related inquiry from another region, and is displayed without erasing either local wording or local context.

## Highlights

- one connected v1.0.0 reference fixture covering observation, AI annotation, Inquiry Draft, publication, transport projection, related inquiry, synthesis, vocabulary mapping, search, replay, and recovery
- deterministic API search and replay validation using FTS5
- mobile/offline Golden Path validation covering local persistence, sensitive-field confirmation, queued synchronization, failure, retry, and Canonical publication
- Conformance Suite v1 with strict Canonical Event validation and an explicit v0.9 compatibility profile
- externally runnable Conformance CLI with machine-readable JSON output
- verified backup manifest, restore, corruption detection, migration plan, dry-run, apply, and derived-state rebuild behavior
- v1 contract index covering Canonical Event, Standard API, workflow, lineage, provenance, transport, AI annotation, and Conformance boundaries
- reference setup and demo guide
- pilot protocol and an explicit record that real participant sessions have not yet been conducted

## Stable boundaries

- Canonical storage is authoritative.
- Raw observations and published Canonical Events remain separately controlled.
- AI output never bypasses human review.
- Annotation acceptance and publication approval are independently attributable decisions.
- Similarity, vocabulary mapping, and semantic relations do not merge Canonical identity.
- Transport failure changes delivery status without invalidating Canonical persistence.
- Search indexes and other projections remain rebuildable derived state.

## Compatibility

The v1 Conformance profile requires the current Canonical Event shape, including `tt:evt:` IDs, `schemaVersion`, `body`, and `provenance.sources`.

Legacy v0.9 fixtures may be checked only through the explicit `v0.9.0` compatibility profile. Legacy aliases are not silently treated as native v1 events.

## Known limitations

- the reference authentication provider is not an OAuth/OIDC-certified deployment
- durable queue and recovery behavior target a single-node reference environment
- external model and transport availability remain optional checks
- lexical and structured FTS5 search is the stable baseline; production RAG, graph inference, and automatic semantic identity merge remain out of scope
- generated, derived, related, translated, or mapped inquiries are not claims of agronomic correctness
- real participant pilot sessions are not yet recorded in the repository

## Upgrade notes from v0.9.0

- run the full workspace test suite before deploying
- run the Conformance CLI against both the v1 reference input and any retained v0.9 compatibility fixtures
- verify backup manifests before upgrade
- execute migrations in plan and dry-run modes before apply
- rebuild search and other derived projections from Canonical storage after restore or migration
- review any integrations that still emit legacy `content`, legacy IDs, or `provenance.rawRef`
