# Toitoi v1.0.0 Release Plan

**Status: preparation** | **Baseline: v0.9.0** | **Started: 2026-07-22** | **Last updated: 2026-07-22**

## Release intent

v1.0.0 completes Toitoi as a reproducible minimum reference implementation of a knowledge space in which inquiries grow from field observations, pass through explicit human review, are published as stable Canonical Events, are rediscovered and derived in other contexts, and retain auditable provenance and lineage throughout the cycle.

This release is an integration, stabilization, evidence, and contract release. It does not introduce a new transport, autonomous AI authority, automatic semantic identity merge, production RAG, graph inference, or internet-scale federation.

## Reference scenario

The v1.0.0 reference scenario is:

> An observation that “only the east side of the field has different weed species” produces human-reviewable inquiry candidates, becomes a published Canonical Event, is connected to a related inquiry from another region, and is displayed in a form that preserves both shared concepts and local context.

The scenario must include:

- original observation
- asynchronous AI inquiry candidates
- annotation acceptance, editing, and rejection
- human-reviewed Inquiry Draft
- independent publication approval
- published Canonical Event
- Nostr, Lingonberry, and ATProto projections
- raw and canonical persistence
- provenance and AI involvement
- lineage and semantic relationships
- context and vocabulary exploration
- lexical and structured search
- replay fixture
- search fixture
- offline and online UI walkthrough
- deterministic end-to-end validation
- operator recovery procedure

## Required scope

### 1. Integrated Golden Path fixture

- create one repository-maintained v1.0.0 reference dataset around the east-side weed observation
- connect observation, annotation, Inquiry Draft, publication, transport projections, related inquiry, derivation, vocabulary mapping, and search results through stable IDs
- preserve local wording while mapping shared concepts explicitly
- include valid, rejected, withdrawn, replayed, and transport-unavailable states where relevant
- make the fixture usable by workspace tests, Conformance Suite, documentation examples, and demo tooling

### 2. Deterministic end-to-end reference flow

Validate the complete flow:

```text
offline observation capture
  → local persistence
  → explicit synchronization
  → asynchronous inquiry generation
  → human annotation review
  → Inquiry Draft promotion and editing
  → independent publication review
  → Canonical persistence
  → multi-transport projection
  → search and context discovery
  → explicit semantic derivation
  → lineage and provenance inspection
  → replay and derived-index rebuild
```

The default CI path must not require a live model, live relay, live PDS, or external network. Existing deterministic providers and fixed transport fixtures remain the CI baseline.

### 3. Stable v1 contracts

Declare and document the following as v1 contracts:

- Canonical Event v1
- Standard API v1
- Inquiry Draft and publication workflow v1
- lineage and semantic relation vocabulary v1
- provenance contract v1
- transport adapter contract v1
- AI annotation contract v1
- Conformance Suite v1

For each contract:

- define the normative document or schema
- identify required and optional fields
- state compatibility and unknown-field behavior
- state authority and human-review boundaries
- identify deprecation and migration rules
- add a machine-verifiable contract or fixture where practical
- avoid stabilizing private implementation details

### 4. Reference implementation packaging

- provide one canonical setup path from repository clone to working reference environment
- provide deterministic demo data initialization
- provide a single command or concise command sequence for starting the API and SPA/PWA
- document optional local-model configuration separately from deterministic mode
- document persistent directories, backup boundaries, and restoration prerequisites
- ensure README navigation reaches setup, demo, specifications, Conformance Suite, operations, governance, and known limitations
- verify commands on a clean environment represented by CI where practical

### 5. Offline and mobile validation

- exercise the Golden Path through the mobile-first SPA/PWA
- verify local observation persistence across restart or reload
- verify explicit synchronization after connectivity recovery
- expose queued, syncing, failed, retryable, reviewed, approved, published, delivered, skipped, and quarantined states
- confirm sensitive information and publication scope before synchronization and publication
- confirm annotation acceptance remains separate from publication approval

### 6. Recovery and external reproducibility

- restore Canonical storage, durable queue state, and configuration from a verified backup
- rebuild search and other derived indexes from Canonical storage
- replay stored inputs and verify equivalent Canonical and derived results
- execute migrations in plan, dry-run, and apply modes
- run Conformance Suite without importing private application modules
- produce machine-readable validation reports suitable for release evidence

### 7. Pilot package and findings

Prepare a small-pilot package for:

- growers or farmers: 2–3 participants
- researcher: 1 participant
- extension or technical practitioner: 1 participant
- developers or operators: 1–2 participants

The repository package must include:

- participant briefing
- informed data and publication boundary
- scenario script
- observation and interview prompts
- task completion record
- issue and safety observation template
- anonymized findings format
- known limitation and follow-up decision format

Repository acceptance requires the pilot protocol and a published findings record. Real participant recruitment and sessions are operational activities outside deterministic CI; absence, substitution, or limitation of the planned pilot must be disclosed explicitly rather than represented as completed evidence.

### 8. Release evidence and documentation

- v1.0.0 release runbook
- v1.0.0 GitHub Release draft
- v1.0.0 release record
- reference scenario walkthrough
- stable-contract index
- pilot protocol and findings record
- known limitations
- upgrade and migration notes from v0.9.0
- final implementation-status update
- synchronized root, API, frontend, operations, governance, and Conformance documentation

## Architectural boundaries

1. Canonical storage remains the source of truth.
2. Search indexes, metrics, queue indexes, and health projections remain rebuildable derived state.
3. Authentication identity, actor identity, transport identity, and Canonical Event identity remain distinct.
4. AI annotation acceptance, relation confirmation, publication approval, moderation, and operator authority remain separate decisions.
5. Similarity, vocabulary mapping, and semantic relations never merge Canonical identity automatically.
6. Raw observation data and externally published Canonical Events remain separately controlled.
7. Transport failure affects delivery status, not Canonical persistence.
8. Offline capture does not imply automatic synchronization or publication.
9. Stable v1 contracts describe observable behavior, not private module layout.
10. Pilot evidence must distinguish observed findings from inferred or simulated results.

## Implementation sequence

```text
v1.0.0 reference fixture and contract inventory
  → integrated deterministic Golden Path
  → mobile/offline E2E completion
  → recovery, replay, migration, and conformance evidence
  → setup and reference-demo packaging
  → pilot protocol and findings record
  → contract freeze and compatibility review
  → release documentation and final regression validation
```

## Acceptance criteria

- [ ] the east-side weed reference scenario is represented by one connected fixture set
- [ ] the complete observation-to-reuse Golden Path passes in default CI
- [ ] offline and online SPA/PWA behavior is covered by deterministic E2E validation
- [ ] AI outputs always pass through an explicit human review boundary
- [ ] annotation acceptance and publication approval remain independently auditable
- [ ] semantic derivation preserves source identity, relation, context, actor, AI involvement, and human confirmation
- [ ] transport projection, re-ingest, and replay preserve Canonical meaning
- [ ] search connects results to original events, provenance, lineage, context, and reuse actions
- [ ] Canonical Event, Standard API, workflow, lineage, provenance, transport, AI annotation, and Conformance contracts are declared as v1
- [ ] compatibility, migration, unknown-field, deprecation, and authority behavior is documented and tested
- [ ] backup, restore, replay, migration, and derived-index rebuild procedures are verified
- [ ] an external environment can run the Conformance Suite and obtain a machine-readable report
- [ ] a clean setup path reaches a working reference API and SPA/PWA
- [ ] at least one documented local-model configuration can generate inquiry candidates without placing AI in the persistence critical path
- [ ] the pilot protocol is published and pilot findings or explicit pilot limitations are recorded
- [ ] known limitations and non-goals are visible from release documentation
- [ ] README navigation reaches setup, demo, specifications, operations, governance, Conformance Suite, and pilot evidence
- [ ] v0.9.0 and all earlier regressions remain green

## Explicitly out of scope

- new transports
- distributed consensus or multi-primary storage
- internet-scale federation
- distributed queue or distributed search engine
- production RAG
- graph inference
- large vector database
- automatic semantic identity merge
- autonomous AI publication or moderation
- generalized social-network features
- claims of agronomic correctness based only on model output
- legal certification for every jurisdiction

## Initial workstreams

### Workstream A: reference fixture and Golden Path

Own the connected scenario data, stable IDs, deterministic orchestration, and complete cross-package E2E test.

### Workstream B: v1 contract freeze

Inventory existing schemas and observable API behavior, close undocumented gaps, and produce the v1 contract index and compatibility tests.

### Workstream C: distribution and operator reproducibility

Validate clean setup, deterministic demo initialization, Conformance execution, backup, restore, replay, migration, and index rebuild.

### Workstream D: mobile/offline release validation

Close remaining SPA/PWA integration gaps and verify visible state transitions, privacy confirmation, retry, and publication boundaries.

### Workstream E: pilot and release evidence

Prepare the pilot package, record findings and limitations, synchronize documentation, and assemble release evidence.

## Release sequence

1. implement required scope on `agent/v1.0.0-reference-implementation`
2. keep the release plan and issue map synchronized with discovered gaps
3. obtain complete green CI and resolve review findings
4. merge the implementation PR
5. synchronize README, implementation status, release notes, contracts, pilot record, and operator documentation
6. validate merged `main` and the clean external setup path
7. create annotated tag `v1.0.0`
8. publish the GitHub Release
9. record released commit, CI run, tag, release URL, pilot evidence, and known limitations
