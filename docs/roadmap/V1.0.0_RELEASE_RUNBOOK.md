# Toitoi v1.0.0 Release Runbook

**Status: implementation** | **Baseline: v0.9.0** | **Target: v1.0.0** | **Last updated: 2026-07-22**

## Purpose

This runbook defines the release gates for Toitoi v1.0.0. A release candidate is acceptable only when the reference scenario, stable contracts, deterministic CI, external reproducibility, recovery procedures, documentation, and pilot evidence or explicit pilot limitations are all recorded.

## Release invariants

The release must preserve the following boundaries:

1. Canonical storage is authoritative.
2. Search indexes, metrics, queue indexes, and health projections are rebuildable derived state.
3. Authentication identity, actor identity, transport identity, and Canonical Event identity remain distinct.
4. AI annotation acceptance, relation confirmation, publication approval, moderation, and operator authority remain separate decisions.
5. Similarity, mapping, and semantic relations never merge Canonical identity automatically.
6. Offline capture does not imply synchronization or publication.
7. Transport failure changes delivery status but does not invalidate Canonical persistence.
8. The deterministic release path does not require live models or external transports.
9. v0.9 compatibility is available only through an explicit compatibility profile.
10. Simulated data and deterministic fixtures are never represented as real pilot findings.

```text

```

- [ ] stable IDs connect observation, annotation, Draft, publication, related inquiry, derivation, vocabulary mapping, transport state, search, and recovery

- [ ] local terms are preserved alongside explicit shared-concept mappings

- [ ] fixture timestamps and identifiers are deterministic

## Phase 3: complete deterministic CI

Run:

```bash
corepack pnpm install --frozen-lockfile
corepack pnpm test
```

Required:

- [ ] complete workspace tests pass
- [ ] v0.9.0 and earlier regressions remain green
- [ ] v1 protocol fixture test passes
- [ ] v1 search and replay test passes
- [ ] v1 frontend offline Golden Path passes
- [ ] v1 backup, restore, and migration test passes
- [ ] Conformance Suite v1 tests pass
- [ ] no test requires network access, live inference, relay credentials, or PDS credentials

Record:

- commit SHA
- PR head SHA
- CI run number and URL
- runner Node.js version
- package manager version

## Phase 5: compatibility verification

Strict v1 input:

```bash
corepack pnpm --filter @toitoi/conformance exec \
  toitoi-conformance \
  --input fixtures/reference/v1.0.0/conformance-input.json \
  --output /tmp/toitoi-v1-conformance.json \
  --pretty
```

Required:

- [ ] exit code is 0
- [ ] `conformanceVersion` is `1.0.0`
- [ ] `compatibilityProfile` is `null`
- [ ] `passed` is `true`
- [ ] `totals.failed` is 0

Legacy compatibility fixture:

```bash
corepack pnpm --filter @toitoi/conformance exec \
  toitoi-conformance \
  --input fixtures/conformance/v0.9.0-transport-round-trips.json \
  --output /tmp/toitoi-v0.9-compatibility.json \
  --pretty
```

- [ ] `conformanceVersion` remains `1.0.0`
- [ ] `compatibilityProfile` is exactly `v0.9.0`
- [ ] legacy acceptance does not weaken strict v1 validation

## Phase 6: offline and mobile Golden Path

- [ ] local observation survives reload or store recreation as supported by the test boundary
- [ ] sensitive location cannot synchronize without explicit acknowledgment
- [ ] queued state remains visible offline
- [ ] synchronization failure becomes visible and retryable
- [ ] retry is initiated explicitly
- [ ] synchronized observation retains the expected Canonical ID
- [ ] annotation acceptance remains separate from publication approval
- [ ] reviewed, approved, published, delivered, skipped, and quarantined states are represented where applicable

## Phase 7: recovery and migration

Verify the reference recovery test and the single-node durability documentation.

- [ ] backup manifest uses relative paths
- [ ] file size and SHA-256 are recorded
- [ ] restore into a different root verifies successfully
- [ ] restored Canonical IDs match the reference manifest
- [ ] queue and audit state are covered as durable state
- [ ] migration plan is deterministic
- [ ] dry-run performs no mutation
- [ ] apply records the resulting version
- [ ] search and other derived state are rebuilt from Canonical storage
- [ ] intentional corruption is detected

## Phase 8: clean setup verification

Follow [`../reference/V1.0.0_SETUP_AND_DEMO.md`](../reference/V1.0.0_SETUP_AND_DEMO.md) in a clean environment.

- [ ] clone and frozen-lockfile installation succeed
- [ ] complete tests pass
- [ ] Conformance CLI succeeds using only the public fixture input
- [ ] API starts with documented persistent directories
- [ ] liveness and readiness endpoints respond correctly
- [ ] authenticated reference mode enforces actor capabilities
- [ ] setup does not depend on unpublished local files

## Phase 9: local model evidence

At least one documented local-model configuration must be tested outside default CI.

- [ ] provider uses the provider-neutral asynchronous AI boundary
- [ ] observation persistence succeeds even when inference fails
- [ ] generated candidates retain model, model version, prompt version, raw output, and timestamps
- [ ] malformed output is rejected
- [ ] generated candidates require human annotation review
- [ ] publication still requires independent approval
- [ ] hardware, model file, quantization, context size, runtime flags, and observed limitations are recorded

A missing live-model validation blocks claiming this criterion as complete but does not justify weakening deterministic CI.

Only after merged-main validation:

```bash

git pull --ff-only
```

- [ ] annotated tag target equals the verified main commit
- [ ] GitHub Release title and body match repository-maintained content
- [ ] release notes include known limitations
- [ ] release links to setup, contracts, Conformance, operations, governance, reference scenario, and pilot evidence
- [ ] public tag and release page are verified

## Stop conditions requiring user judgment

Pause and request a decision only when one of the following occurs:

- a proposed change breaks an already published external contract
- a security or privacy boundary requires policy selection rather than implementation clarification
- real pilot recruitment, participant consent, or publication of findings requires authorization
- release criteria cannot be satisfied without changing declared v1 scope
- a tag or public GitHub Release is ready to be published

All other implementation, test, documentation, and review work should proceed without interruption.
