# Toitoi v1.0.0 Roadmap

**Status: active** | **Baseline: v0.4.0** | **Target: v1.0.0** | **Last updated: 2026-07-17**

## 目的

この文書は、v0.4.0以降のToitoiを、実験的な基盤実装から「問いが育つ知識空間」の最小参照実装へ到達させるための、リリース単位の工程を定義します。

Toitoiはv0.4.0までに、Canonical Event、multi-transport、storage / replay、Standard API、identity、provenance、lineage、Inquiry Draft、human review、publication guard、非同期AI annotation layerを接続しました。

今後の中心課題は、新しい抽象層やtransportを増やすことではありません。既存の基盤を、現場の観察から問いの生成、公開、発見、派生、再利用まで続く一つの利用体験へ統合することです。

中長期の思想と改善原則は[`IMPROVEMENT_ROADMAP.md`](./IMPROVEMENT_ROADMAP.md)、AI導入の安全境界と段階は[`AI_ADOPTION_ROADMAP.md`](./AI_ADOPTION_ROADMAP.md)を正本とします。この文書は、それらをv1.0.0までのrelease sequenceへ落とし込む役割を持ちます。

---

## v1.0.0における「一応の完成」

v1.0.0は、将来構想をすべて実装した版ではありません。次のGolden Pathを、一般利用者向けUI、再現可能なテスト、運用手順、参照データを含めて完了した最小参照実装と定義します。

```text
現場で観察を記録する
  ↓
AIが問い候補を提示する
  ↓
人間が確認・修正する
  ↓
Canonical Eventとして保存・公開する
  ↓
別の利用者が発見する
  ↓
翻訳・注釈・派生・統合する
  ↓
問いの系譜と来歴を確認する
  ↓
新しい観察や判断材料として現場へ還流する
```

最低限、次の状態を満たしたとき、Toitoiを「一応完成した」と判断します。

- 一人の利用者が観察から問いを作成し、人間確認後に公開できる
- 別の利用者がその問いを検索・発見し、意味関係を明示して派生できる
- lineage、context、provenance、AI関与、人間確認状態を追跡できる
- スマートフォンと不安定通信環境を含む参照フローが成立する
- 外部の開発者がセットアップ、fixture、Conformance Suiteから挙動を再現できる
- 小規模な実地パイロットで、利用価値と安全境界を確認している

## 現在地: v0.4.0

v0.4.0では、保存済みCanonical Eventの後段に非同期AI補助層を接続しました。

```text
stored observation / inquiry
  → asynchronous AI job
  → auditable summary / tag annotation
  → human acceptance
  → derived Inquiry Draft
  → human review
  → approved Canonical Event publication
```

v0.4.0時点で、次の基盤は成立しています。

- protocol-independent Canonical Event
- Nostr / Lingonberry / ATProtoのmulti-transport boundary
- raw / canonical append-only storageとdeterministic replay
- Standard APIとtransport-independent frontend model
- canonical identity、provenance、lineage
- Inquiry Draft、human review、publication guard
- lineage tree、context exploration
- duplicate-aware AI job queue、retry、recovery、annotation provenance
- accepted annotationからderived Inquiry Draftへのpromotion

一方、次は未完成です。

- production inference providerと実ローカルモデル
- inquiry-generation annotation contract
- annotation review / promotion mutation API
- 完成したSPA / PWA
- オフライン保存と同期
- 利用者向け派生操作
- 実用的な横断検索と語彙管理
- production authentication / authorization / rate limiting
- Conformance Suite、schema migration tooling、共同運営手続き
- 実地パイロット

## v0.5.0: 実用的なAI問い生成

### 目的

v0.4.0の要約・タグannotation基盤を、Toitoiの中心機能である問い候補生成へ拡張します。

### 必須範囲

- llama.cppまたは互換ローカル推論APIへ接続するproduction inference provider
- provider-neutral boundaryとdeterministic CI providerの維持
- inquiry-generation annotation schema
- 一つの観察から複数の問い候補を生成する契約
- `inquiry`、`context`、`observation`、`relationship`、`uncertainty`、`tags`、`source_refs`の保持
- model、model version、prompt version、raw output、生成日時のprovenance
- malformed outputのschema rejection
- annotationのaccept / edit / reject mutation API
- accepted annotationからInquiry Draftを作成する操作
- 4GB RAMクラスを想定した低資源実行プロファイル
- AI品質を農業上の正しさと同一視しないreview boundary

### Golden Path

```text
observationを保存
  → 非同期AI jobを作成
  → 実モデルが複数の問い候補を生成
  → 人間が候補を編集・採用
  → lineage付きInquiry Draftを作成
  → publication guardが未承認公開を拒否
```

### 完了条件

- 実ローカルモデルで決定的に検証可能なfixtureがある
- inference failureがingest / persistenceを停止しない
- 生成理由、入力元、モデル、prompt版、raw outputを追跡できる
- 人間が修正・却下できる
- accepted annotationとpublication approvalが混同されない

### 対象外

- AI生成物の無人公開
- RAG
- semantic identity merge
- 大規模distributed queue

## v0.6.0: SPA / PWAとオフラインGolden Path

### 目的

既存API、review model、rendererを、一般利用者が一つのアプリケーションから操作できる体験へ統合します。

### 必須範囲

- 観察入力画面
- AI問い候補レビュー画面
- Inquiry Draft編集とhuman review画面
- 公開前確認とpublication result表示
- inquiry detail、provenance、lineage、context画面
- job、annotation、同期状態の表示
- PWA installability
- IndexedDB等によるローカル観察保存
- offline publish queue
- 接続回復後の明示的同期
- raw observationと外部公開Canonical Eventの分離
- 位置情報、個人情報、非公開文脈の公開前確認
- スマートフォンを基準とした入力操作

### Golden Path

```text
圃場で観察
  → 圏外でローカル保存
  → 端末またはローカルサービスで問い候補生成
  → 人間が確認
  → 接続回復
  → 公開
  → 公開結果とprovenanceを確認
```

### 完了条件

- スマートフォンだけで主要操作を完了できる
- offline / online遷移でデータを失わない
- 同期失敗と再試行状態を利用者が確認できる
- 公開対象を人間が最終確認できる
- Golden Path E2E testがUI境界を含めて再現する

## v0.7.0: 問いの派生と系譜

### 目的

inquiryを単なる投稿ではなく、翻訳、注釈、対照、再解釈、統合を通じて成長する知識単位として扱います。

### 必須範囲

次のsemantic relationを第一級操作として扱います。

- `derived_from`
- `translated_from`
- `observed_alongside`
- `contrasts_with`
- `synthesizes`
- `reframes`
- `annotates`
- `revises`

加えて、次を実装します。

- 既存inquiryから派生Inquiry Draftを作成するUI
- relation種別に応じた入力補助とvalidation
- lineage treeの系譜ビュー
- contexts / relationshipsを中心とする文脈ビュー
- AIによるrelation候補と人間による確定の区別
- provenance card
- canonical identity、semantic relation、context similarityの視覚的分離
- transport再取得後もlineageを復元できるE2E fixture

### 完了条件

- 別の利用者が既存の問いを派生できる
- 派生元、意味関係、作成者、AI関与、人間確認を追跡できる
- 類似性だけを根拠にidentity mergeしない
- lineage treeから元の観察と公開イベントへ遡れる

## v0.8.0: 検索・語彙・再利用

### 目的

蓄積された問いを、単語、文脈、地域、意味関係から再発見し、別の現場へ再利用できるようにします。

### 必須範囲

- SQLite FTS5によるinquiry / observation / summary / tags横断検索
- context、relation、transport、provenance、review stateによるfilter
- 地域、気候、土壌、作物、季節などのcontext exploration
- 類似inquiry候補
- embeddingを利用する場合のidentity判定との分離
- Core Vocabulary
- Domain Vocabulary
- Local Vocabulary
- Local Vocabularyと共有概念を接続するmapping / translation claim
- 検索結果から派生操作へ進むUI
- reference datasetを利用した検索品質評価

### 導入順

```text
FTS5
  → structured filters
  → context / relation exploration
  → similarity candidates
  → optional embeddings
```

### 完了条件

- 別地域の関連inquiryを地域語を失わずに発見できる
- exact identityとrelated candidateをUI / APIで区別できる
- 検索結果が元イベント、provenance、lineageへ接続される
- 検索品質を固定fixtureで回帰検証できる

### 対象外

- embeddingによる自動identity merge
- 大規模search engineへの早期移行
- graph inference
- production RAG

## v0.9.0: 共同運用・安全性・外部互換性

### 目的

個人の開発環境から、小規模な共同利用と外部実装に耐えられる状態へ引き上げます。

### 必須範囲

#### APIと権限

- authentication
- authorization
- rate limiting
- actor / user identity
- review、publish、moderation権限
- audit log
- pagination
- idempotency
- stable error contract

#### 運用

- durable job queueまたは明示された単一ノード耐久構成
- dead-letter handling
- graceful shutdown
- backup / restore
- replay runbook
- schema migration tooling
- health check
- structured logsとmetrics
- transport別delivery status

#### Conformance Suite

- JSON Schema validation
- required fieldsとschema version検証
- Canonical ID維持試験
- provenance / rawRef責務分離
- unknown fieldsとmigration試験
- Nostr / Lingonberry / ATProtoのsemantic round trip
- live ingestとreplayの同値性
- 共通fixture

#### ガバナンス

- Canonical Event変更手続き
- schema versioningと互換性方針
- TIPの提出、議論、採択方法
- vocabulary追加・変更手続き
- deprecated fieldの扱い
- moderation原則
- 地域知、伝統知、共同体知の権利保護
- 位置情報、個人情報、秘密情報の扱い
- 訂正、撤回、削除要求とappend-only storageの関係

### 完了条件

- 複数の利用者を権限分離して運用できる
- 障害後にstorage、queue、derived indexを復元できる
- 外部実装がConformance Suiteで互換性を検証できる
- 運営者と利用者の責任境界が文書化されている

## v1.0.0: 最小参照実装の完成

### 目的

Toitoi固有の価値を、一つの再現可能な参照ユースケース、実地パイロット、安定契約として確定します。

### 参照ユースケース

最初のv1.0.0 reference scenarioは、次を推奨します。

> 「畑の東側だけ雑草の種類が違う」という観察から問いが生成され、別地域の類似した問いと接続され、比較可能な形で表示される。

このシナリオに、少なくとも次を揃えます。

- original observation
- AI inquiry candidates
- human-reviewed Inquiry Draft
- published Canonical Event
- transport projections
- raw / canonical persistence
- provenance
- lineage / relationships
- context exploration
- replay fixture
- search fixture
- UI walkthrough
- deterministic E2E test
- operator runbook

### 実地パイロット

小規模なパイロットグループを構成します。

```text
農家・栽培者       2〜3名
研究者             1名
普及員・技術者     1名
開発者             1〜2名
```

確認項目:

- 観察入力の負担
- 問い候補が観察や思考を狭めていないか
- context抽象化によって重要情報を失っていないか
- 他地域の問いが実際に参考になるか
- provenanceとAI関与を理解できるか
- lineage treeが実用的か
- 非公開情報を確実に除外できるか
- 訂正、撤回、再公開の操作が理解できるか

### 安定契約

v1.0.0では、少なくとも次をv1 contractとして宣言します。

- Canonical Event v1
- Standard API v1
- Inquiry Draft / publication workflow v1
- lineage / semantic relation vocabulary v1
- provenance contract v1
- transport adapter contract v1
- AI annotation contract v1
- Conformance Suite v1

### Release criteria

- reference scenarioのGolden Pathがdefault CIで再現する
- offline / onlineを含むUI E2Eが成功する
- 一つ以上の実ローカルモデル構成で問い生成が動作する
- AI出力は常にhuman review boundaryを通る
- transport projectionを再取得・再生して意味が維持される
- backup / restore / replay / migration手順が検証済みである
- Conformance Suiteを外部環境から実行できる
- pilot findingsと既知の制約が公開されている
- READMEからセットアップ、デモ、仕様、運用文書へ到達できる

## v1.0.0まで優先しない事項

次は、Golden Pathと実地利用が安定した後の候補とします。

- 新しいtransportの追加
- 大規模federation
- graph inference
- 高度なRAG
- 大規模vector database
- 大規模distributed search engine
- semantic similarityによる自動identity merge
- AIによる無人公開
- 汎用SNS機能

これらが特定のrelease acceptance criteriaを満たすために必要となった場合のみ、範囲を再評価します。

## 横断的な判断原則

すべてのrelease scopeと設計変更を、次の順で評価します。

1. 問いの循環を強化するか
2. 人間の観察力と探究心を増幅するか
3. 地域固有の文脈を失わせないか
4. raw data、位置情報、個人情報を過剰に集約しないか
5. Canonical Eventの意味的安定性を維持できるか
6. provenance、lineage、replayを損なわないか
7. 特定transport、AI provider、事業者への依存を増やさないか
8. 低資源・不安定通信環境でも利用可能か
9. 外部実装が検証可能か
10. コモンズとして持続可能か

## 成功指標

### 問いの循環

- observationからinquiryが作成された割合
- AI候補の採用、編集、却下割合
- inquiryから派生inquiryが作成された割合
- 翻訳、annotation、reframe、synthesisへ発展した件数

### 文脈と多様性

- 複数contextをまたいで接続されたinquiry数
- Local Vocabularyが保持された割合
- 異なる地域から追加されたobservation数

### 信頼性

- provenanceを追跡可能なevent割合
- replayで再現可能なevent割合
- Conformance Suite適合率
- ambiguous identity mergeの件数

### 現場適合性

- offline環境で完了できた操作の割合
- 4GB RAMクラス端末での処理時間
- 観察入力と公開確認に要する時間
- 公開前に利用者が内容と公開範囲を確認できた割合

## 結論

v0.4.0までに、Toitoiの難しい基盤設計の大部分は成立しました。v1.0.0までの中心課題は、その基盤を利用者体験へ統合し、問いが現場から生まれ、他者の文脈で翻訳・派生され、系譜を持って成長し、再び現場へ戻る循環を実証することです。

> Toitoiでなければ成立しない「問いが育つ知識空間」を、再現可能な最小参照実装として完成させる。
